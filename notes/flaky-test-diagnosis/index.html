Flaky test 진단 — 4가지 원인과 격리 패턴

Flaky test는 보통 시간·상태·환경·외부 의존 4가지 원인. 어디서 깨졌는지 모르면 retry로 가린다. 원인별 진단 + 격리 패턴.

  • #testing
  • #ci
  • #diagnosis
  • #quality

Flaky test가 나쁜 이유

같은 코드가 어떨 땐 통과, 어떨 땐 실패. 처음엔 "retry로 해결" 하지만 누적되면:

  • CI 신뢰도 0 — 실패해도 "또 flaky?" 무시
  • 진짜 버그가 flaky 사이에 숨음
  • developer 시간 손실 (재실행 대기)

retry는 증상 가림. 원인 진단이 핵심.

원인 1 — 시간 의존 (Time)

가장 흔한 원인.

// 깨지기 쉬움
test('expires after 1 hour', async () => {
  const token = createToken();
  await sleep(3600 * 1000); // 1시간 대기? 불가
  expect(token.isExpired()).toBe(true);
});

// 또 다른 패턴
test('processes within 100ms', async () => {
  const start = Date.now();
  await process();
  expect(Date.now() - start).toBeLessThan(100); // 머신 속도 영향
});

진단 신호:

  • 로컬에선 통과, CI에선 실패 (CI 머신이 느림)
  • 같은 머신에서 부하 시 실패
  • setTimeout·sleep·Date.now·performance.now 사용

해결:

  • 시간을 mock: jest.useFakeTimers(), sinon.useFakeTimers()
  • 시간 의존 로직을 함수로 분리: function isExpired(now, expiresAt) → now를 주입
  • 절대 시간(< 100ms) 대신 결정론적 검증

원인 2 — 상태 누수 (State Leak)

테스트 간 상태가 새는 경우.

// 깨지기 쉬움
let counter = 0;

test('A', () => {
  counter++;
  expect(counter).toBe(1); // 단독 실행만 통과
});

test('B', () => {
  counter++;
  expect(counter).toBe(1); // A가 먼저 실행되면 2 = FAIL
});

또는 DB·파일시스템·전역 객체 누수.

진단 신호:

  • 테스트 순서 바꾸면 결과 다름 (--shuffle 옵션)
  • 병렬 실행 시 실패 (직렬에선 통과)
  • 단독 실행 통과, 다른 테스트와 같이 실행 시 실패

해결:

  • beforeEach로 상태 초기화
  • 전역 mock은 afterEach로 cleanup
  • DB 테스트는 트랜잭션 + rollback 또는 schema reset
  • 테스트 간 순서 의존 금지

원인 3 — 환경 의존 (Environment)

특정 환경에서만 동작.

test('reads config', () => {
  const config = loadConfig(); // process.env 사용
  expect(config.apiKey).toBeDefined(); // CI에 env var 누락 시 FAIL
});

test('formats date', () => {
  expect(formatDate(new Date())).toBe('2026-05-29'); // 머신 timezone 영향
});

진단 신호:

  • timezone·locale·OS 따라 결과 다름
  • env var 누락 시 실패
  • 네트워크 환경 (proxy·VPN·firewall) 영향

해결:

  • env var은 테스트 fixture에 명시 (디폴트 + override)
  • timezone·locale을 명시: process.env.TZ = 'UTC'
  • 파일시스템 경로는 path.join (cross-OS)

원인 4 — 외부 의존 (External)

DB·API·외부 서비스 호출.

test('fetches user', async () => {
  const user = await fetch('/api/user/1'); // 외부 API
  expect(user.name).toBe('Alice');
});

진단 신호:

  • 네트워크 끊김 시 실패
  • 외부 서비스 응답 시간 변동 시 timeout
  • 외부 서비스 데이터 변경 시 결과 다름
  • rate limit 걸림

해결:

  • 단위 테스트는 외부 의존 mock (MSW·nock)
  • 통합 테스트는 격리된 환경 (docker-compose)
  • E2E는 staging API 또는 mocked production

격리 패턴

flaky를 발견하면 즉시 해결 못 할 수도. 임시 격리:

// 격리 1: skip + 이슈 등록
test.skip('flaky-test-name', () => { ... }); // → 이슈 트래커에 등록 필수

// 격리 2: retry (마지막 수단)
test('flaky', async () => { ... }, { retry: 3 });

// 격리 3: 별도 파일로 분리
// __tests__/flaky/auth.test.js → CI에서 별도 stage

중요: 격리는 임시. 1주일 이내에 원인 분석 + 수정 또는 삭제.

진단 워크플로우

flaky 발견
  ↓
1. 100회 반복 실행 (실패율 측정)
  ↓
2. 단독 실행 vs 다른 테스트와 같이 실행 비교 (상태 누수 진단)
  ↓
3. 순서 셔플 (--shuffle) (순서 의존 진단)
  ↓
4. CI vs 로컬 비교 (환경·시간 의존 진단)
  ↓
5. 외부 의존 mock 후 재실행 (외부 의존 진단)
  ↓
원인 분류 → 위 4가지 해결책

함정

  • retry로 가리기: 가장 흔한 anti-pattern. 진짜 버그가 retry 사이에 숨음.
  • flaky를 "예전부터 있던 것"으로 무시: 누적되면 CI 전체 신뢰도 0.
  • 격리 후 안 돌아옴: skip한 테스트가 영원히 skip. 격리는 1주일 한도.
  • 단위 테스트가 외부 의존: 단위 테스트는 외부 의존 0. 외부 의존 있으면 통합 테스트.
  • 테스트가 random 데이터 생성: faker·random()이 가끔 edge case 만나면 실패. seed 고정 또는 deterministic fixture.

핵심

Flaky test는 4가지 원인(시간·상태·환경·외부) 중 하나. retry로 가리기 전에 원인 진단이 핵심. 원인을 알면 해결책이 정해져 있다.

관련 노트

  • 🌳 TypeScript의 거짓 안전감 — 타입이 통과해도 버그는 남는다

    TypeScript 통과 = 안전이라는 직관. 그러나 런타임 데이터·외부 API·any 의 escape hatch는 컴파일러가 못 본다. 타입은 안전의 일부일 뿐.

  • 🌱 커버리지 갭 분석 — 강한 곳 말고 빈 곳을 측정으로 찾기

    사람은 잘하는 영역에 더 추가하고 빈 영역은 못 본다. 작업물의 분포를 측정하면 어디가 비었는지 객관적으로 드러난다. 테스트·콘텐츠·문서·스킬 포트폴리오 공통.

  • 🌱 모호한 지시를 측정 기준으로 — "잘 해줘"를 실행 가능하게

    "전체적으로 개선해줘", "다양하게", "깔끔하게"는 실행자마다 다르게 해석된다. 작업 전에 모호한 품질 단어를 측정 가능한 기준으로 번역하면 재작업과 오해가 줄어든다.

  • 🌱 같은 함정에 두 번 빠지면 체크리스트가 된다

    한 번의 실수는 수정으로 끝나지만, 같은 실수가 두 번 반복되면 그건 개인의 부주의가 아니라 시스템의 공백이다. 함정의 재발 횟수에 따라 대응을 격상하는 사다리.

  • 🌱 CI 캐시·incremental build — 빌드 30분을 5분으로

    monorepo CI 시간은 모든 PR의 비용. 캐시 레이어·incremental build·병렬화로 절감. 작업 시간이 아니라 cumulative cost로 보면 절감 가치가 압도적.
