lastTendedAt으로 사이트가 살아있음을 보여주기

게시일(publishedAt) 대신 최종 다듬은 날(lastTendedAt)을 표시하면, 방문자에게 "이 사이트는 계속 자라고 있다"는 신호가 직접 전달된다.

  • #garden
  • #writing
  • #content-architecture

일반 블로그의 함정

블로그 글에는 publishedAt이 있다. 시간이 지나면 "오래된 글"로 보인다. 2년 전 글은 자동으로 신뢰 ↓.

문제: 글의 내용은 계속 다듬어지는데도, 사이트는 "2023-01-15"라는 한 시점에 박혀 있다.

디지털 가든의 해법

plantedAt (처음 심은 날) + lastTendedAt (최근 다듬은 날). 둘 다 표시.

plantedAt: 2026-04-12  # 처음 심음
lastTendedAt: 2026-05-19  # 어제 다듬음

방문자가 보는 것:

  • "2026-04-12 심음, 2026-05-19 마지막 다듬음"
  • → "이 노트는 한 달간 계속 자라고 있다"
  • → "이 사이트는 살아있다"

정렬 효과

lastTendedAt을 sort key로 쓰면, 최근에 다듬어진 노트가 상단. 게시일 기준 정렬은 "1년 전 글이 영원히 1년 전 글"이지만, 다듬은 날 기준은 "현재 관심사가 자연스럽게 상단".

함정

  • 매번 lastTendedAt 수동 업데이트 = 부담. git commit timestamp로 자동화 가능.
  • "다듬었다"의 기준 모호 — 오타 수정도 카운트? 본문 5문장 추가는?
  • 너무 자주 lastTendedAt 갱신 = 노이즈. 의미 있는 다듬기에만.

핵심

사이트의 "살아있음"은 게시 빈도가 아니라 다듬기 빈도다.

관련 노트

  • 🌳 Incrementally correct — 한 번에 완성하지 않는 사이트

    Brian Lovin이 제안한 개념. 글이 "출간"되는 게 아니라 시간을 두고 점진적으로 정확해지는 운영 방식.

  • 🌳 콘텐츠 = 파일 — MDX + Vite의 자동 확장 패턴

    콘텐츠 한 개 = MDX 파일 한 개. import.meta.glob으로 자동 인덱싱하면 코드 수정 없이 사이트가 확장된다.

  • 🌿 디지털 가든은 왜 블로그보다 운영 부담이 적은가

    시간순 블로그는 "이번 달은 안 썼네"가 누적되지만, 디지털 가든은 노트의 성숙도만 다듬어도 사이트가 살아있게 됩니다.

  • 🌱 PR 본문은 코드보다 먼저 읽힌다

    리뷰어는 코드를 보기 전에 PR 본문을 본다. 본문이 비어있거나 "WIP"이면 리뷰어는 코드를 추측해서 읽는다. 본문 5분 투자로 리뷰 30분 절약.

  • 🌱 Note 라운드 패턴 — 신규 → cross-link → 함정 → 다음 라운드

    노트 50편을 모은 게 한 번에 일어난 게 아니다. 신규 양산 → 누적 후 cross-link → 함정 섹션 보강 → 다음 라운드의 4단계 사이클이 반복됐다.
