테스트 vs 타입 — 둘 다 안 한다면 어디부터

두 가지 모두 가치 있지만, 시간 제약 시 우선순위가 갈린다. 외부 경계와 비즈니스 룰은 테스트, 내부 함수 시그니처는 타입. 둘은 보완 관계.

  • #testing
  • #typescript
  • #swe

질문

새 프로젝트 시작. 시간 제한. 둘 중 어디부터?

  • 타입 (TypeScript 도입)
  • 테스트 (Jest·Vitest 도입)

답은 "둘 다", "상황에 따라" 같은 정답 회피가 아니다. 각자 잘 잡는 영역이 다르다.

잘 잡는 영역

타입 (TypeScript)

  • 내부 함수 시그니처: (a: User, b: Order) => Receipt
  • 리팩토링 안전성: 이름 바꿔도 호출자 자동 추적
  • 자동완성·IDE 도움: 개발 속도 ↑
  • 타입 자체가 문서: 함수 호출 시 시그니처가 곧 사용법

테스트

  • 비즈니스 룰: "주말은 영업일이 아니다"
  • 엣지 케이스: null/undefined·빈 배열·overflow
  • 외부 통신: API·DB·파일 시스템
  • 회귀 방지: 한 번 잡은 버그가 다시 안 나옴

우선순위 가이드

상황우선
단일 함수 라이브러리 (npm 패키지)타입 우선 — 사용자 IDE 친화
비즈니스 로직 중심 SaaS테스트 우선 — 룰 변화에 회귀
외부 API 통합 많음테스트 우선 — 통합 케이스 잡기
팀 새로 합류, 코드베이스 큼타입 우선 — 학습 곡선 ↓
빠른 prototype둘 다 미루기 OK, MVP 후 도입

함께 쓰면 더 강함

// 타입: 시그니처
function calculateReceipt(order: Order): Receipt { ... }

// 테스트: 룰
test('weekend orders get +10% surcharge', () => {
  expect(calculateReceipt(weekendOrder).total).toBe(110);
});

타입은 컴파일 시점, 테스트는 런타임. 둘이 다른 시점에서 다른 종류를 잡는다.

둘 다 안 하는 위험

  • 작은 변경이 멀리서 깨짐 (회귀)
  • "여기 호출하면 뭐 들어와?" 매번 확인
  • 새 팀원 학습 곡선 가파름
  • 6개월 후 본인이 모름

함정

  • "둘 다 안 함 + 나중에 도입": prototype 단계에서 미루기 시작하면 영원히 안 됨. MVP 직후가 최적.
  • 타입 통과 = 안전하다는 거짓 신뢰: 타입은 내부 일관성. 외부 데이터·런타임은 못 잡음.
  • 테스트를 위한 추상화 과잉: 의존성 주입·인터페이스 분리를 테스트만을 위해 → 본 코드 복잡도 ↑. 테스트하기 어렵다면 신호이지 강제가 아님.
  • 늦게 도입: 1만 줄 코드에 타입 도입은 한 사람 몫의 작업. 작을 때 도입이 비용 최소.

핵심

타입은 내부 일관성, 테스트는 외부 약속. 둘은 같은 문제를 다른 시점에서 잡는다.

관련 노트

  • 🌳 TypeScript의 거짓 안전감 — 타입이 통과해도 버그는 남는다

    TypeScript 통과 = 안전이라는 직관. 그러나 런타임 데이터·외부 API·any 의 escape hatch는 컴파일러가 못 본다. 타입은 안전의 일부일 뿐.

  • 🌱 커버리지 갭 분석 — 강한 곳 말고 빈 곳을 측정으로 찾기

    사람은 잘하는 영역에 더 추가하고 빈 영역은 못 본다. 작업물의 분포를 측정하면 어디가 비었는지 객관적으로 드러난다. 테스트·콘텐츠·문서·스킬 포트폴리오 공통.

  • 🌱 안 쓰는 의존성의 숨은 비용 — install·lockfile·transitive CVE·supply chain

    package.json에 있는데 import 0건인 패키지는 단순 dead code가 아니다. install 시간·lockfile 부피·transitive CVE·supply chain 표면적이 모두 누적된다. 정리는 정기 작업.

  • 🌱 Flaky test 진단 — 4가지 원인과 격리 패턴

    Flaky test는 보통 시간·상태·환경·외부 의존 4가지 원인. 어디서 깨졌는지 모르면 retry로 가린다. 원인별 진단 + 격리 패턴.

  • 🌱 야간 장애의 두 단계 — 임시 패치 vs 근본 수정

    새벽에 장애 알람을 받으면 "지금 멈춰야 한다 vs 제대로 고쳐야 한다"가 충돌한다. 두 단계로 분리하면 의사결정이 빨라진다.
