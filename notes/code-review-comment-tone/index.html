코드 리뷰 코멘트의 톤 — 비판이 아니라 질문으로

같은 지적도 "이건 틀렸어요"와 "이 부분 의도가 궁금해요"의 결과가 다르다. 톤은 단순 매너가 아니라 리뷰의 정확도와 팀의 학습 속도를 결정한다.

같은 지적의 두 버전

버전 A: "여기서 useMemo 잘못 쓰셨네요. dependency가 매 렌더마다 바뀌어서 메모이제이션이 의미 없어요."

버전 B: "이 useMemo의 dependency가 매 렌더마다 새 객체일 것 같은데, 의도하신 게 맞나요?"

두 버전이 같은 사실을 가리키는데 결과는 다르다.

  • A: 작성자가 "내가 틀렸나" → 방어 → 빠르게 수정하거나 반박. 학습 안 됨
  • B: 작성자가 "내가 생각 못 한 게 있나" → 의도 설명 → 두 사람이 같이 발견. 학습됨

가끔 B가 잘못된 지적일 수도 있다 (작성자가 의도적으로 한 선택). A로 적으면 "틀렸다"는 단정이 먼저 나가서 잘못 지적했을 때 후퇴가 어렵다. B로 적으면 의문이 먼저 나가서 자연스럽게 두 사람이 답을 만든다.

톤이 결정하는 것

톤은 매너가 아니다. 리뷰의 정확도를 결정한다.

  • 단정형: 작성자가 의도를 설명할 자리가 닫힘 → 리뷰어가 본 단편만 정답이 됨
  • 질문형: 작성자가 컨텍스트를 보여줄 자리가 열림 → 두 사람의 정보가 합쳐짐

코드 리뷰의 목적이 "틀린 부분 잡기"가 아니라 "더 좋은 코드 만들기"라면, 후자가 더 자주 더 좋은 결과를 낸다.

4가지 톤 패턴

1. 질문형 (Question)

가장 안전하고 가장 학습이 큰 톤.

  • "이건 왜 이렇게 하셨어요?"
  • "다른 옵션도 고려하셨나요? 예를 들어 X는요?"
  • "이 부분 동작이 궁금한데 설명해주실 수 있나요?"

2. 제안형 (Suggestion)

답을 안다고 생각하지만 명령은 아닌 톤.

  • "이거 X 패턴으로 바꾸면 어떨까요?"
  • "X를 고려해봤는데 어떻게 생각하세요?"
  • nit: prefix로 "선택사항이지만" 신호 보내기

3. 정보형 (Information)

지적이 아니라 정보 제공.

  • "참고로 이 케이스에서 X 라이브러리가 정확히 같은 일을 해요"
  • "예전에 비슷한 자리에서 X 함정에 빠진 적이 있어요. 여기는 괜찮을까요?"

4. 단정형 (Assertion)

명확히 틀렸을 때만. 보안 취약점·데이터 손실·production 장애 위험 같은 경우.

  • "이건 SQL injection 위험입니다. 반드시 parameterized query로."
  • "이 시점에 transaction을 안 닫으면 connection leak."

단정형은 톤이 강한 만큼 틀리면 안 된다. 100% 확신할 때만.

우선순위

같은 PR에 5개 지적이 있다면, 80%는 질문형/제안형. 단정형은 진짜 막아야 하는 1-2개만.

질문형으로도 강조 가능하다. 톤이 부드럽다고 신호가 약한 게 아니다. "이거 의도하신 게 맞나요?"는 작성자가 자기 코드를 다시 보게 만드는 강한 신호다.

함정

  • "솔직함"으로 단정형 남발: 솔직함은 정확도와 무관. 잘못된 단정형은 솔직한 게 아니라 그냥 틀린 거다.
  • passive-aggressive 질문: "이게 정말 맞나요???"처럼 톤이 비꼬는 질문형은 단정형보다 더 안 좋다. 진짜 궁금증인지 자문.
  • 모든 지적에 nit 붙이기: 진짜 중요한 지적에도 nit이 붙으면 중요도 신호가 사라진다. 진짜 nit만 nit.
  • 시니어의 단정형 디폴트: 시니어가 단정형으로 적으면 주니어는 반박할 자리가 없어진다. 같은 지적도 시니어일수록 질문형으로.
  • 외국 회사 톤 그대로: 한국어 직역 "이건 안 좋습니다"는 영어 "this isn't great"보다 훨씬 강하게 들린다. 언어별 톤 보정 필요.

핵심

톤은 매너가 아니라 리뷰의 정확도와 팀 학습 속도를 결정한다. 80%는 질문형/제안형, 20%만 단정형. 시니어일수록 질문형 비중을 높인다.

관련 노트

  • 🌱 AI 반복 위임의 mode collapse — 출력이 한 점으로 수렴한다

    AI에게 "더 만들어줘"를 반복하면 출력이 점점 비슷해진다. 모델이 고확률 영역으로 수렴하는 mode collapse다. 다양성은 저절로 안 나오고 명시적으로 강제해야 한다.

  • 🌱 모호한 지시를 측정 기준으로 — "잘 해줘"를 실행 가능하게

    "전체적으로 개선해줘", "다양하게", "깔끔하게"는 실행자마다 다르게 해석된다. 작업 전에 모호한 품질 단어를 측정 가능한 기준으로 번역하면 재작업과 오해가 줄어든다.

  • 🌱 비동기 PR 리뷰 SLO — 팀의 throughput을 결정하는 숫자

    PR 리뷰가 며칠씩 걸리는 팀은 throughput이 절반이다. 리뷰 SLO를 명시하면 팀의 lead time이 비선형으로 짧아진다. SLO 설계와 함정.

  • 🌿 코드 리뷰의 진짜 가치 — 코드보다 결정

    코드 리뷰에서 가장 가치 있는 코멘트는 변수명·스타일 지적이 아니라 "이 결정의 이유는?"이다. 코드는 자명하면 충분, 결정은 자명하지 않으면 영영 잃는다.

  • 🌱 1on1을 잘 쓰는 5가지 — 매니저·매니지 양쪽이 시간을 잘 쓰는 법

    1on1이 status report로 끝나면 시간 낭비다. 매니저·매니지 양쪽이 시간을 잘 쓰는 5가지 패턴. 1on1은 양쪽이 쓰는 자리지 매니저가 매니지를 쓰는 자리가 아니다.
