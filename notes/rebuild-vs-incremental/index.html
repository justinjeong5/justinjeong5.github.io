리빌드 vs 점진 개선 — 사이트는 리빌드, 콘텐츠는 점진

같은 손으로 같은 사이트를 리빌드는 1일에, 콘텐츠는 3일에 걸쳐 점진으로 추가했다. 두 모드를 의도적으로 분리한 이유와 언제 어느 쪽을 고르는지.

  • #decision
  • #refactoring
  • #sustained-delivery
  • #writing

같은 사이트, 다른 두 모드

5/19에 사이트를 통째로 리빌드했다. 라우팅·MDX·디자인 토큰·모바일 GNB·인터랙티브 레이어·빌드 청크. 한 세션에서 다.

5/20-5/21에는 같은 사이트에 콘텐츠를 추가했다. Note 라운드 2 + 16편 cross-link + 야간장애 시리즈 + 홈레이더 케이스 + 개인 레포 발견 9편. 3일에 걸쳐 작은 PR/커밋 묶음으로.

같은 손, 같은 사이트, 다른 두 모드. 의도적이었다.

리빌드가 맞는 자리

구조가 바뀌는 작업은 리빌드가 안전하다.

  • 라우팅 모델이 바뀜 (SPA → MDX 기반 + 디지털 가든)
  • 디자인 토큰 single source 도입 (CSS 변수)
  • 빌드 청크 분할 정책 (manualChunks)

이런 건 점진으로 가면 중간 상태가 깨진다. 라우팅을 절반만 바꾸면 사이트가 동작 안 한다. 디자인 토큰을 절반만 바꾸면 색이 섞인다. 그래서 한 번에 끝낸다.

리빌드의 신호:

  • 변경이 여러 레이어를 동시에 건드림
  • 중간 상태가 사용자에게 노출되면 안 됨
  • 기존 코드 일부를 그대로 두기 어려움 (의존성 그래프가 통째로)

점진이 맞는 자리

추가가 누적되는 작업은 점진이 안전하다.

  • 노트 1편 추가 → 사이트 그대로 동작
  • 케이스 1편 추가 → 기존 케이스에 영향 0
  • cross-link 추가 → 기존 노트의 본문 변경 없음 (frontmatter lastTendedAt만 갱신)

점진의 신호:

  • 변경이 새 파일/엔트리 추가가 중심
  • 중간 상태(글 N편)도 사용자에게 의미 있음
  • 한 번에 끝낼 필요가 없음 — 매주 1편씩이 더 자연스러움

헷갈리는 자리

frontmatter 스키마를 바꾸는 작업이 헷갈린다. growth 필드 추가는?

  • 점진: 새 노트부터 적용, 옛 노트는 그대로
  • 리빌드: 50편 전부 일괄 업데이트

리더 친화도가 결정한다. 옛 글에 growth가 없으면 카드 뱃지가 비어있다 — 사용자가 "이 글은 왜 뱃지가 없지?"라고 묻는다. 그 인지 부담이 크면 리빌드. 작으면 점진.

이 사이트에서는 리빌드를 선택했다. growth/lastTendedAt이 가든 hygiene의 핵심 신호여서 누락된 글은 약한 신호가 된다.

한 PR 안에 섞으면

리빌드와 점진을 한 PR에 섞으면 리뷰가 안 끝난다.

  • 리빌드 변경 = diff가 큼 (구조 변경)
  • 점진 변경 = diff가 작음 (파일 추가)

섞이면 리뷰어가 "구조 변경이 정말 모든 새 파일에 적용됐나?"를 확인해야 한다. 분리하면 구조 변경은 구조 변경끼리, 추가는 추가끼리 본다.

→ PR 분리는 모드 분리부터.

함정

  • 점진을 리빌드로 처리: 노트 1편 추가에 사이트 전체 회귀 테스트를 돌리면 시간만 든다.
  • 리빌드를 점진으로 처리: 라우팅 절반만 바꾸고 PR을 올리면 동작 안 한다.
  • 모드를 시간으로 결정: "오늘은 시간 있으니까 리빌드"는 위험. 작업 성격이 결정한다.
  • 모드 전환 흔적 안 남김: 리빌드 PR과 점진 PR의 커밋 메시지/태그가 다르지 않으면 나중에 git log에서 안 보임. feat(rebuild): vs feat(content):처럼 prefix 다르게.

핵심

같은 손이라도 구조는 리빌드, 추가는 점진. 모드를 섞지 말고, 모드를 결정한 다음 도구를 고른다.

관련 노트

  • 🌳 Incrementally correct — 한 번에 완성하지 않는 사이트

    Brian Lovin이 제안한 개념. 글이 "출간"되는 게 아니라 시간을 두고 점진적으로 정확해지는 운영 방식.

  • 🌿 A/B 테스트 멈춤 신호 — 통계적 유의성 도달 전에 끝내야 할 때

    A/B 테스트는 '유의성 도달까지 기다리기'가 디폴트로 알려져 있지만, 실무에서는 유의성 도달 전에 멈춰야 하는 신호가 더 자주 발생한다. 5가지 신호와 멈춤 결정 프레임.

  • 🌿 디지털 가든은 왜 블로그보다 운영 부담이 적은가

    시간순 블로그는 "이번 달은 안 썼네"가 누적되지만, 디지털 가든은 노트의 성숙도만 다듬어도 사이트가 살아있게 됩니다.

  • 🌱 PR 본문은 코드보다 먼저 읽힌다

    리뷰어는 코드를 보기 전에 PR 본문을 본다. 본문이 비어있거나 "WIP"이면 리뷰어는 코드를 추측해서 읽는다. 본문 5분 투자로 리뷰 30분 절약.

  • 🌱 Note 라운드 패턴 — 신규 → cross-link → 함정 → 다음 라운드

    노트 50편을 모은 게 한 번에 일어난 게 아니다. 신규 양산 → 누적 후 cross-link → 함정 섹션 보강 → 다음 라운드의 4단계 사이클이 반복됐다.
