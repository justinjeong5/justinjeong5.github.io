OAuth state·PKCE 함정 — 흐름의 보안 디테일

OAuth authorization code flow는 단순해 보이지만 state·PKCE·redirect URI 검증을 빼먹으면 CSRF·토큰 가로채기·open redirect 취약점. 5가지 함정.

  • #oauth
  • #auth
  • #web

OAuth 흐름 (간단 버전)

1. 클라이언트 → /authorize?client_id=X&redirect_uri=Y&state=Z
2. 사용자 로그인 + 권한 동의
3. /authorize → redirect to Y?code=ABC&state=Z
4. 클라이언트 → /token (code=ABC) → access_token

각 단계에 함정이 있다.

함정 1 — state 누락 (CSRF)

state 파라미터는 클라이언트가 생성한 무작위 값. 흐름 시작 시 보내고, callback에서 같은 값이 돌아오는지 검증.

state 없으면: 공격자가 자기 OAuth 흐름을 시작해서 사용자에게 redirect URL을 보내면, 사용자 브라우저가 공격자의 계정으로 연결됨.

// 흐름 시작
const state = crypto.randomUUID();
sessionStorage.setItem('oauth_state', state);
location.href = `/authorize?...&state=${state}`;

// callback
const returnedState = new URL(location.href).searchParams.get('state');
const expectedState = sessionStorage.getItem('oauth_state');
if (returnedState !== expectedState) {
  throw new Error('CSRF detected');
}

state는 반드시 검증. 누락하면 OAuth가 CSRF에 취약.

함정 2 — PKCE 없음 (SPA·모바일)

전통적 OAuth는 client_secret으로 인증. 그러나 SPA·모바일은 secret을 안전하게 저장 못 함.

PKCE (Proof Key for Code Exchange): secret 없이 인증.

1. code_verifier = random(64자)
2. code_challenge = SHA256(code_verifier)  
3. /authorize?...&code_challenge=X&code_challenge_method=S256
4. /token (code=ABC&code_verifier=원본) → 서버가 SHA256(verifier) == challenge 검증

attacker가 code를 가로채도 code_verifier가 없어 토큰 교환 불가.

모든 public client(SPA·mobile·desktop)는 PKCE 필수. 2025년 이후 OAuth 2.1 표준에선 confidential client도 권장.

함정 3 — redirect_uri 검증 부족

공격자가 redirect_uri를 자기 도메인으로 바꾸면 code가 공격자에게 전달된다.

서버 측 검증 필요:

  • 정확한 일치 (substring·prefix 검증 부족)
  • HTTPS 강제 (http:// 거부)
  • 사전 등록된 URI만 허용

잘못된 검증:

# 취약 — substring 검증
if "myapp.com" in redirect_uri: # → "myapp.com.attacker.com" 통과
    accept()

# 안전 — 정확 일치
if redirect_uri in REGISTERED_URIS:
    accept()

함정 4 — implicit flow 사용

옛 OAuth의 implicit flow는 redirect URL에 access token을 직접 담는다.

/callback#access_token=abc

문제:

  • token이 browser history·referrer·로그에 노출
  • refresh token 없음
  • replay 공격에 취약

OAuth 2.1에서 implicit flow는 deprecated. authorization code + PKCE 사용.

함정 5 — token을 URL fragment에 두기

implicit flow뿐 아니라 callback URL에 token을 fragment(#...)로 두는 모든 패턴이 위험.

/callback#token=abc
  • 브라우저 history에 남음
  • Service Worker가 fetch URL을 로그할 수 있음
  • analytics 도구가 fragment를 캡처할 수 있음

token은 POST body 또는 Authorization header로만 전달.

OAuth 2.1 흐름 (권장 — 2025+)

1. 클라이언트 → code_verifier 생성
2. /authorize?response_type=code&code_challenge=X&state=Y
3. 사용자 로그인
4. /callback?code=ABC&state=Y (state 검증)
5. POST /token (code, code_verifier) → access_token + refresh_token
6. access_token은 Authorization: Bearer header로 사용
7. refresh: POST /token (refresh_token=X) → 새 access + 새 refresh (rotation)

각 단계에 위 함정 5개를 다 막아야 함.

함정 (메타)

  • OAuth 라이브러리 그대로 신뢰: 라이브러리도 misconfig 가능. state 검증·PKCE·redirect 검증을 직접 확인.
  • token revocation 무시: 사용자가 권한 철회해도 발급된 access token은 만료까지 유효. 짧은 exp + refresh 필수.
  • 권한(scope) 너무 넓게 요청: 필요 최소 scope만. 과한 scope는 사용자 거부율 ↑.
  • error 응답 정보 노출: error_description에 client_secret·user info 노출 사고 자주.
  • CORS preflight를 OAuth endpoint에 안 함: 브라우저 CORS 에러로 흐름 중단.

핵심

OAuth는 흐름은 단순하지만 state·PKCE·redirect_uri·token 전달 위치 4가지 디테일이 보안의 전부다. 라이브러리 쓰더라도 이 4가지는 직접 검증.
