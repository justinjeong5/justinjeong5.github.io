exactly-once 전달의 환상 — at-least-once + 멱등성이 답

메시지 큐에서 "정확히 한 번 전달"은 대부분 마케팅 용어다. 네트워크에서 진짜 exactly-once는 불가능에 가깝다. at-least-once 전달 + 소비자 멱등성 = 실질적 한 번.

  • #distributed
  • #data
  • #messaging
  • #backend

세 가지 전달 보장

메시지 큐(Kafka·SQS·RabbitMQ)는 세 가지 전달 보장을 말한다.

  • at-most-once: 최대 한 번. 손실 가능, 중복 없음.
  • at-least-once: 최소 한 번. 손실 없음, 중복 가능.
  • exactly-once: 정확히 한 번. 손실도 중복도 없음.

대부분 exactly-once를 원한다. 그런데 진짜 exactly-once는 분산 환경에서 거의 불가능하다.

왜 exactly-once가 어려운가

핵심은 메시지 처리와 ack(확인 응답)가 원자적이지 않다는 것.

1. 소비자가 메시지 수신
2. 메시지 처리 (DB 쓰기)
3. 큐에 ack 전송

2와 3 사이에 소비자가 죽으면? 처리는 됐는데 ack가 안 갔다. 큐는 메시지를 다시 보낸다 → 중복 처리.

ack를 먼저 보내면(3→2)? ack 후 처리 전에 죽으면 → 메시지 손실.

처리와 ack를 원자적으로 묶을 수 없으니, 손실 아니면 중복 중 하나를 택해야 한다. 그래서 "exactly-once"는 시스템 경계 안에서의 환상이고, 경계를 넘으면 깨진다.

"exactly-once"라 부르는 것들의 실체

상용 시스템이 말하는 exactly-once는 대부분:

  • Kafka exactly-once: 프로듀서 idempotence + 트랜잭션. 단 Kafka 내부(Kafka→Kafka)에서만. 외부 DB로 나가면 깨짐.
  • Flink/Spark exactly-once: checkpoint + 상태 복원. 외부 sink가 멱등하거나 트랜잭셔널해야 성립.

공통점: 경계 안에서만 exactly-once. 외부 시스템(DB·API·결제)으로 나가는 순간 at-least-once + 멱등성으로 돌아온다.

실용 답 — at-least-once + 멱등성

진짜 해법은 단순하다.

  1. 전달은 at-least-once로 (손실 없음 보장)
  2. 소비자를 멱등하게 (중복이 와도 결과 동일)
def consume(message):
    if store.seen(message.id):   # 이미 처리?
        return                   # 중복 무시
    process(message)
    store.mark_seen(message.id)

중복이 와도 seen 체크로 흡수. 손실은 at-least-once가 막는다. 결과적으로 effectively-once.

이게 멱등성 키 패턴과 정확히 같다. 메시지 ID가 멱등성 키.

outbox 패턴 — DB와 큐의 원자성

처리(DB 쓰기)와 메시지 발행을 원자적으로 묶고 싶을 때.

문제: DB 커밋 후 큐 발행 사이에 죽으면 → 메시지 유실

outbox 패턴:

  1. 비즈니스 데이터 + outbox 레코드를 같은 DB 트랜잭션으로 커밋
  2. 별도 프로세스가 outbox를 폴링해서 큐로 발행
  3. 발행 성공하면 outbox 레코드 삭제(또는 표시)

DB 트랜잭션이 원자성을 보장하고, outbox 폴링이 at-least-once 발행을 보장. 소비자 멱등성이 중복을 흡수.

광고 이벤트 파이프라인에 적용

광고 클릭/노출 이벤트는 대량 + 손실 민감(정산).

  • 수집: at-least-once (이벤트 손실 = 매출 손실)
  • 소비: event_id 멱등성으로 dedup
  • 정산: 윈도우 마감 시 최종 dedup (eventual하게 중복 제거)

실시간 집계는 약간의 중복 허용, 정산 직전에 강한 dedup. 강한 일관성이 필요한 지점에만 비용 지불.

함정

  • "exactly-once" 마케팅을 그대로 믿기: 경계 안에서만. 외부 sink로 나가면 멱등성 필요.
  • at-most-once를 무의식 선택: ack를 처리 전에 보내면 손실. 돈 걸린 이벤트엔 치명적.
  • 멱등성 없이 at-least-once: 중복이 그대로 통과. 집계 2배·과금 2배.
  • seen 체크 스토리지 무한: 메시지 ID 저장도 TTL/윈도우 필요.
  • outbox 폴링 지연 무시: outbox는 즉시 발행이 아니라 폴링 지연 있음. 실시간성이 필요하면 트레이드오프 인지.
  • 순서 보장과 혼동: exactly-once ≠ 순서 보장. 순서가 필요하면 파티션 키 별도 설계.

핵심

분산 환경에서 진짜 exactly-once는 시스템 경계 안의 환상이다. 실용 답은 at-least-once 전달 + 소비자 멱등성 = effectively-once. 멱등성이 분산 데이터 정합성의 핵심 도구다.
