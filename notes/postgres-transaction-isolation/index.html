Postgres transaction isolation — 4 레벨이 실제로 다루는 것

READ UNCOMMITTED·READ COMMITTED·REPEATABLE READ·SERIALIZABLE 4 레벨. Postgres에서 실제로 동작이 다른 건 3개다. 각 레벨이 막는 anomaly와 비용.

  • #database
  • #postgres

왜 isolation을 고민하나

동시 트랜잭션이 같은 데이터를 만지면 4가지 anomaly가 발생한다.

  • Dirty Read: 커밋 안 된 변경을 읽음
  • Non-repeatable Read: 같은 행을 두 번 읽었는데 값이 다름
  • Phantom Read: 같은 쿼리를 두 번 실행했는데 row 집합이 다름
  • Serialization Anomaly: 동시 실행 결과가 어떤 직렬 순서로도 안 나옴

isolation level은 어떤 anomaly를 막을지 결정한다. 강할수록 안전하지만 처리량 ↓.

Postgres 특이점:

  • READ UNCOMMITTED는 실제로 READ COMMITTED처럼 동작 (구분 안 함)
  • REPEATABLE READ가 표준보다 강함 — Phantom Read도 막음 (snapshot isolation)
  • → 실제로 다른 레벨 3개: READ COMMITTED / REPEATABLE READ / SERIALIZABLE

디폴트 — READ COMMITTED

Postgres 디폴트. 대부분의 작업이 여기서 동작.

BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- 100
-- (다른 트랜잭션이 balance를 200으로 변경 + 커밋)
SELECT balance FROM accounts WHERE id = 1; -- 200 (변경 보임)
COMMIT;

같은 트랜잭션 안에서 같은 행을 두 번 읽으면 다른 값이 나올 수 있다.

언제 충분한가:

  • 단일 행 업데이트 (예: counter++)
  • 짧은 트랜잭션
  • 일관성보다 처리량 우선

언제 부족한가:

  • 같은 트랜잭션에서 여러 행을 일관되게 읽어야 할 때
  • 보고서 생성 (다른 행이 트랜잭션 중간에 변경되면 보고가 inconsistent)

REPEATABLE READ — Snapshot Isolation

트랜잭션 시작 시점의 snapshot을 읽음. 같은 트랜잭션 안의 SELECT는 항상 같은 결과.

BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1; -- 100
-- (다른 트랜잭션이 200으로 변경 + 커밋)
SELECT balance FROM accounts WHERE id = 1; -- 100 (snapshot 유지)
COMMIT;

언제 쓰는가:

  • 보고서 / 백업 (일관된 시점의 데이터)
  • 멀티 row 읽기를 일관되게

함정 — Write Skew: REPEATABLE READ는 Write Skew를 막지 못한다.

-- 의사 1, 의사 2가 동시에 oncall 사퇴 시도
-- 규칙: 최소 1명은 oncall이어야 함

T1: SELECT COUNT(*) FROM oncall WHERE active = true; -- 2
T2: SELECT COUNT(*) FROM oncall WHERE active = true; -- 2
T1: UPDATE oncall SET active = false WHERE id = 1; -- "나만 빠지면 1명 남음"
T2: UPDATE oncall SET active = false WHERE id = 2; -- "나만 빠지면 1명 남음"
T1: COMMIT;
T2: COMMIT;
-- 결과: 0명 oncall (둘 다 잘못된 가정으로 행동)

각 트랜잭션은 자기 snapshot 기준으론 옳지만, 합쳐서 보면 규칙 위반.

SERIALIZABLE — Write Skew도 막음

Postgres의 SERIALIZABLE은 Serializable Snapshot Isolation (SSI) 알고리즘 사용. 동시 트랜잭션 간 의존성을 추적해서 비직렬화 가능 패턴을 감지하면 한 트랜잭션을 abort.

위 예시에서 T2가 commit 시도하면 ERROR: could not serialize access 발생. 애플리케이션이 retry해야 함.

언제 쓰는가:

  • 복잡한 invariant (Write Skew 위험)
  • 금융 트랜잭션
  • 정합성이 처리량보다 절대 우선

비용:

  • abort + retry가 자주 일어남 (특히 핫 row)
  • 트랜잭션 의존성 추적 메모리 ↑
  • 처리량 보통 30-50% ↓

실용 선택 가이드

default: READ COMMITTED
  ↓ 같은 트랜잭션 안에서 일관된 읽기가 필요?
REPEATABLE READ
  ↓ Write Skew 위험이 있는 invariant?
SERIALIZABLE

또는 SERIALIZABLE 대신:

  • SELECT FOR UPDATE (명시적 lock) — 핫 row가 적으면 lock이 SSI보다 가벼울 수 있음
  • Advisory lock — 애플리케이션 수준 mutex
  • 비관적 → 낙관적 (version column + retry)

함정

  • 트랜잭션 안 쓰고 isolation 걱정: autocommit 모드에선 isolation 의미 없음. BEGIN으로 명시적 트랜잭션 필요.
  • REPEATABLE READ로 Write Skew 막을 거라 착각: 못 막는다. Write Skew는 SERIALIZABLE 또는 lock 필요.
  • SERIALIZABLE 디폴트로: 처리량 30-50% 손실. 정말 필요한 트랜잭션에만.
  • abort 후 retry 안 함: SERIALIZABLE은 abort가 정상 동작. 애플리케이션이 retry 로직 필수.
  • long transaction: REPEATABLE READ에서 트랜잭션이 길면 VACUUM이 row 정리 못 함 → table bloat.
  • read-only 트랜잭션도 isolation 필요: 읽기만 해도 inconsistent snapshot이 영향. SET TRANSACTION READ ONLY로 명시.

핵심

Postgres에서 실제로 다른 isolation 레벨은 3개. READ COMMITTED가 디폴트, 일관된 멀티 row 읽기는 REPEATABLE READ, Write Skew 위험은 SERIALIZABLE 또는 명시적 lock. SERIALIZABLE은 디폴트가 아니다 — 비용이 크니 필요한 곳에만.
