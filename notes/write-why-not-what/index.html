코드 주석에는 "왜"를 적어라 — 6개월 후의 자기를 위해

코드는 "무엇"을 한다. 주석은 "왜"를 적어야 한다. 6개월 후의 자기 또는 다른 사람이 이해해야 한다.

  • #code-quality
  • #writing
  • #swe

흔한 주석

// 사용자 수를 가져온다
const count = users.length;

이 주석은 0의 가치. 코드 자체가 이미 "사용자 수"를 명확히 말한다. 이런 주석을 안 적는 게 낫다 — 노이즈만 만들고 코드와 주석이 어긋날 위험.

가치 있는 주석

// users 배열이 null인 경우는 OAuth 콜백 직후 race condition.
// 처음에 ?? 0으로 처리했으나 그러면 카운트가 잘못 표시되어
// 명시적 throw로 호출자가 처리하도록 함 (#1234)
const count = users?.length ?? (() => { throw new RaceConditionError(); })();

이 주석은:

  • 왜 이 처리가 필요한지 (race condition)
  • 왜 다른 방법 안 썼는지 (?? 0의 실패)
  • 링크 (#1234)로 더 깊은 컨텍스트

어떤 "왜"인가

  • 비즈니스 룰의 이유 ("주말은 영업일이 아님")
  • 직관과 다른 코드의 이유 ("성능 때문에 일부러 비효율적인 알고리즘")
  • 하지 않은 것의 이유 ("재시도 로직 추가 안 함 — idempotent하지 않음")
  • 트레이드오프 ("정확성 vs 응답속도에서 응답속도 선택")

핵심 원칙

미래의 자기 또는 다른 사람이 "왜 이렇게 했지?"라고 물을 가능성이 있으면, 그 답을 미리 써둔다.

코드는 시간이 지나면 컨텍스트가 사라진다. 주석은 컨텍스트를 보존한다.

함정

  • 모든 줄에 "왜" 주석 = 노이즈. 놀라움이 있는 코드에만.
  • "왜"가 바뀌면 주석도 갱신해야 함. 안 하면 오해 유발.
  • 진짜 "왜"는 commit message + PR description + linked issue에 잘 분산. 주석은 코드 옆에 두는 보조.

적용

읽기 좋은 코드 = (자명한 코드) + (자명하지 않은 부분의 "왜" 주석). 둘 다 필요.

관련 노트

  • 🌳 Incrementally correct — 한 번에 완성하지 않는 사이트

    Brian Lovin이 제안한 개념. 글이 "출간"되는 게 아니라 시간을 두고 점진적으로 정확해지는 운영 방식.

  • 🌳 TypeScript의 거짓 안전감 — 타입이 통과해도 버그는 남는다

    TypeScript 통과 = 안전이라는 직관. 그러나 런타임 데이터·외부 API·any 의 escape hatch는 컴파일러가 못 본다. 타입은 안전의 일부일 뿐.

  • 🌱 안 쓰는 의존성의 숨은 비용 — install·lockfile·transitive CVE·supply chain

    package.json에 있는데 import 0건인 패키지는 단순 dead code가 아니다. install 시간·lockfile 부피·transitive CVE·supply chain 표면적이 모두 누적된다. 정리는 정기 작업.

  • 🌱 야간 장애의 두 단계 — 임시 패치 vs 근본 수정

    새벽에 장애 알람을 받으면 "지금 멈춰야 한다 vs 제대로 고쳐야 한다"가 충돌한다. 두 단계로 분리하면 의사결정이 빨라진다.

  • 🌿 npm 패키지 출시 체크리스트 — bin·files·engines·dry-run

    npm publish 한 번이 영구다. bin 경로·files glob·engines·--dry-run·SemVer 6가지를 사전 검증. 실수 한 줄이 100명의 install error를 만든다.
