N+1 쿼리 — ORM이 만들고 ORM이 못 잡는 함정

1개의 쿼리로 끝날 일이 N+1번 도는 패턴. ORM의 추상화가 자동으로 만들고, 같은 ORM이 자동으로 못 잡는다. detect는 도구로, fix는 패턴으로.

  • #database
  • #backend
  • #performance
  • #orm

문제 — 1번의 의도가 N+1번이 되는 순간

ORM(Prisma·TypeORM·SQLAlchemy 등)으로 다음 코드를 작성한다.

const orders = await prisma.order.findMany({ where: { status: 'paid' } });
for (const order of orders) {
  const user = await prisma.user.findUnique({ where: { id: order.userId } });
  console.log(user.email);
}

의도: "paid 주문들과 각 주문의 사용자 이메일을 가져온다." 한 번의 join으로 끝날 일이다.

실제 실행: orders 1번 + user N번 = N+1 쿼리. orders가 100건이면 101번의 DB roundtrip. 1000건이면 1001번. p99 latency가 비선형으로 폭증한다.

왜 못 보는가

  • 코드는 깔끔하다. 명령형 for 루프 + ORM 호출
  • 로컬 개발 DB는 작아서 N+1이 빨라 보임 (100ms도 안 걸림)
  • 단위 테스트는 mock이라 쿼리 수 검증 안 함
  • production에서 100ms × 1000건 = 100초. 이때 발견

detect — 도구로

  • ORM 쿼리 로깅: 한 요청 당 쿼리 수 카운트
  • APM trace (Datadog·Sentry Performance): 같은 endpoint의 쿼리 수 비교
  • 테스트 assertion: "이 endpoint는 N건 가져와도 5개 이하 쿼리"
  • EXPLAIN ANALYZE: ORM이 생성한 SQL 직접 확인

도구 없이 본 코드만 읽어서 N+1을 잡는 건 매우 어렵다. 추상화 안에 숨어있다.

fix — 패턴으로

패턴 1: include / select (eager loading)

const orders = await prisma.order.findMany({
  where: { status: 'paid' },
  include: { user: { select: { email: true } } },
});

한 번의 join으로 끝남.

패턴 2: in 절 일괄

const orders = await prisma.order.findMany({ where: { status: 'paid' } });
const userIds = orders.map(o => o.userId);
const users = await prisma.user.findMany({ where: { id: { in: userIds } } });

2번의 쿼리. join이 어려운 케이스에서 활용.

패턴 3: dataloader (GraphQL·복잡한 graph)

같은 요청 내에서 자동 batching. graphql 서버에서 흔히 씀.

함정

  • include로 끝났다고 믿기: include는 한 단계만. user → posts → comments까지 들어가면 또 N+1 가능.
  • map 안에 await 다 풀어버리기: await Promise.all(orders.map(...))는 여전히 N개 쿼리. 병렬일 뿐 1개로 줄지 않음.
  • production에서만 재현: 로컬 DB는 작아서 안 드러남. staging에 production-like 데이터 필요.
  • fix 후 EXPLAIN 안 봄: ORM이 만든 join이 inefficient한 경우. 큰 테이블끼리 join하면 index 안 맞아 더 느릴 수 있음.

회귀 방지 — 쿼리 수를 테스트로 고정

N+1은 한 번 고쳐도 다음 PR에서 쉽게 재발한다. 누가 include를 빼거나 루프 안에 쿼리를 추가하면 조용히 N+1로 돌아온다. 코드 리뷰로는 안 잡힌다(추상화에 숨어서).

→ 쿼리 수를 assertion으로 고정한다.

test('주문 목록은 쿼리 2개 이하', async () => {
  const queries = await captureQueries(() =>
    getOrdersWithUsers({ status: 'paid' })
  );
  expect(queries.length).toBeLessThanOrEqual(2);
});

이게 flaky test 진단과 짝이다 — 쿼리 수 테스트가 없으면 N+1 회귀가 "가끔 느린 endpoint"라는 flaky 증상으로만 나타나고 원인 진단이 어렵다. 결정론적 쿼리 수 검증이 회귀를 PR에서 막는다.

핵심

ORM은 N+1을 자동으로 만들고 자동으로 못 잡는다. detect는 도구, fix는 패턴으로 따로 도입해야 한다.

관련 노트

  • 🌱 광고 지표 함정 — CTR·CVR·viewability·정산 불일치

    광고 플랫폼의 지표는 정의가 미묘해서 같은 이름이 양쪽에서 다른 값이 된다. CTR 분모·viewability 기준·conversion 윈도우·정산 dedup. 4가지 함정.

  • 🌱 exactly-once 전달의 환상 — at-least-once + 멱등성이 답

    메시지 큐에서 "정확히 한 번 전달"은 대부분 마케팅 용어다. 네트워크에서 진짜 exactly-once는 불가능에 가깝다. at-least-once 전달 + 소비자 멱등성 = 실질적 한 번.

  • 🌱 Frequency capping을 분산 환경에서 — 정확성 vs 비용

    "사용자에게 같은 광고를 하루 3번까지"는 단일 서버면 쉽다. 수십 대 서버 + 초당 수만 요청이면 정확한 카운팅이 비싸다. 정확성과 비용의 트레이드오프 4단계.

  • 🌱 멱등성 키 — 중복 과금·이벤트를 막는 설계

    네트워크는 재시도하고 재시도는 중복을 만든다. 정산·결제·광고 이벤트에서 중복은 돈 문제다. 멱등성 키의 설계 — 키 생성·저장·윈도우·응답 재현.

  • 🌱 JWT vs session — 스토리지·만료·revocation의 트레이드오프

    JWT가 stateless라 좋다는 말은 절반만 맞다. revocation·만료·스토리지를 같이 보면 session이 더 단순한 경우가 많다. 5가지 차원 비교.
