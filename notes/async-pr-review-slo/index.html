비동기 PR 리뷰 SLO — 팀의 throughput을 결정하는 숫자

PR 리뷰가 며칠씩 걸리는 팀은 throughput이 절반이다. 리뷰 SLO를 명시하면 팀의 lead time이 비선형으로 짧아진다. SLO 설계와 함정.

리뷰 대기가 throughput을 죽인다

PR 1개의 lead time을 분해하면 보통 이렇다.

  • 코드 작성: 4시간
  • PR 열기: 5분
  • 첫 리뷰까지 대기: 1-3일 ← 여기
  • 리뷰 코멘트 대응: 1시간
  • 두 번째 리뷰까지 대기: 1-2일 ← 여기
  • 머지: 5분

코드 작성보다 리뷰 대기가 lead time의 80%를 차지한다. PR 1개당 3-5일이 표준이 된다.

이걸 SLO로 바꾸면 lead time이 비선형으로 짧아진다.

SLO 예시

팀에서 합의한 숫자.

항목목표
첫 리뷰까지평일 기준 4시간 이내 (P50), 24시간 이내 (P95)
코멘트 대응 후 재리뷰2시간 이내 (P50)
머지 후 deploy까지30분 이내 (CI 통과 시)

숫자가 크지 않다. 일을 더 시키는 게 아니라 순서를 바꾸는 것이다.

왜 효과가 비선형인가

PR 1개의 lead time이 5일 → 1일로 줄면 lead time만 5배 짧아지는 게 아니다.

WIP(work in progress)이 같이 줄어든다. 5일짜리 PR이 동시에 N개 열려있는 상태에서, 1일짜리로 줄면 동시 열린 PR이 N/5개로 줄어든다. 작성자는 자기 PR이 빨리 머지되니까 다음 작업으로 빨리 옮길 수 있다.

리뷰 품질도 같이 올라간다. 1일된 PR과 5일된 PR을 리뷰하는데 같은 시간이 든다고 가정해도, 1일된 PR이 컨텍스트가 더 선명해서 리뷰 정확도가 높다. 5일 전에 작성된 PR을 리뷰하면 작성자도 컨텍스트가 흐려져서 review 응답이 느려진다.

운영 규칙

SLO를 설정한다고 자동으로 지켜지진 않는다. 운영 메커니즘 필요.

1. 리뷰 notification 정리

Slack/Discord에 PR 알림 채널 분리. 멤버는 그 채널만 보고 자기 차례인 리뷰를 빠르게 본다.

2. round-robin assignee

CODEOWNERS + GitHub auto-assign. 작성자가 "누구 리뷰 부탁" 안 골라도 자동 배정.

3. 리뷰 우선순위 명시

  • 🔥 hotfix → 30분 이내
  • 🚀 작은 변경 (< 200 lines) → 4시간 이내
  • 📦 큰 변경 (> 500 lines) → 24시간 이내 (그리고 나누자고 제안)

4. async pairing time

매일 30분-1시간을 "리뷰 전용 시간"으로 캘린더 블록. 그 시간에 쌓인 리뷰를 한 번에 처리.

5. SLO 위반 모니터링

월별로 SLO 위반 빈도 측정. 위반이 많으면 SLO를 낮추거나 (현실 인정) 운영을 바꾸거나 (인력 추가·교육).

SLO ≠ 더 빨리 일하기

오해 1: SLO는 사람을 더 빨리 일하게 만드는 도구가 아니다.

오해 2: SLO는 리뷰 품질을 낮추는 도구가 아니다.

SLO의 핵심은 순서다. "지금 작성한 코드보다 리뷰 대기 중인 PR이 더 lead time critical하다"는 우선순위를 명시한 것. 본인 일을 잠시 멈추고 리뷰를 먼저 처리하는 게 팀 throughput에 더 기여한다.

이게 동의 안 되면 SLO가 안 지켜진다. SLO는 합의 위에 동작.

함정

  • SLO 없이 "빨리 봐주세요": 매번 작성자가 리뷰어를 졸라야 하는 운영. 작성자·리뷰어 양쪽 비용. SLO는 명시적 규칙.
  • 너무 짧은 SLO: 1시간 이내 첫 리뷰는 깊은 작업 흐름을 깬다. 4시간이 적당.
  • SLO 위반 페널티: 위반 시 누구 잘못 따지는 운영은 사람 부담만 늘림. 위반 빈도 모니터링이지 개인 책임 추궁이 아님.
  • 모든 PR에 같은 SLO: hotfix와 큰 리팩토링이 같은 SLO면 둘 다 잘못 처리됨. 우선순위별 SLO 분리.
  • 시니어가 리뷰 안 함: 시니어가 SLO 위반의 주범이면 팀이 안 따라옴. 시니어부터 SLO 지키기.
  • non-blocking 리뷰 vs blocking 리뷰 혼동: nit 코멘트는 머지를 막지 않아야 한다. 단정형 지적만 blocking.

핵심

리뷰 대기가 lead time의 80%다. SLO 4-24시간으로 명시하면 throughput이 비선형으로 올라간다. 운영 메커니즘(알림 채널·round-robin·전용 시간) + 합의(작성자·리뷰어 양쪽 우선순위 동의)가 SLO를 동작시킨다.

관련 노트

  • 🌱 AI 반복 위임의 mode collapse — 출력이 한 점으로 수렴한다

    AI에게 "더 만들어줘"를 반복하면 출력이 점점 비슷해진다. 모델이 고확률 영역으로 수렴하는 mode collapse다. 다양성은 저절로 안 나오고 명시적으로 강제해야 한다.

  • 🌱 코드 리뷰 코멘트의 톤 — 비판이 아니라 질문으로

    같은 지적도 "이건 틀렸어요"와 "이 부분 의도가 궁금해요"의 결과가 다르다. 톤은 단순 매너가 아니라 리뷰의 정확도와 팀의 학습 속도를 결정한다.

  • 🌱 1on1을 잘 쓰는 5가지 — 매니저·매니지 양쪽이 시간을 잘 쓰는 법

    1on1이 status report로 끝나면 시간 낭비다. 매니저·매니지 양쪽이 시간을 잘 쓰는 5가지 패턴. 1on1은 양쪽이 쓰는 자리지 매니저가 매니지를 쓰는 자리가 아니다.

  • 🌱 PR 본문은 코드보다 먼저 읽힌다

    리뷰어는 코드를 보기 전에 PR 본문을 본다. 본문이 비어있거나 "WIP"이면 리뷰어는 코드를 추측해서 읽는다. 본문 5분 투자로 리뷰 30분 절약.

  • 🌱 CodeRabbit + Cubic 이중 자동 리뷰 — 노이즈인가 신호인가

    PR마다 CodeRabbit과 Cubic AI가 둘 다 리뷰를 단다. 두 자동 리뷰를 같이 쓰는 게 가치인지 노이즈인지를 며칠 써본 후의 관찰.
