🌿Budding

작게 배포하고 다시 읽기 — 한 사이클에 다 하지 않기

큰 변경을 한 번에 하려는 유혹이 가장 큰 위험. 작은 사이클로 쪼개 매 사이클마다 빌드·배포·사용자 반응 확인.

  • #workflow
  • #deploy
  • #design-process

원칙

큰 변경(디자인 리뉴얼, 모바일 최적화, 콘텐츠 추가)을 한 번에 다 하려 하면:

  • 머지 충돌 ↑
  • 버그 한 번에 누적 → 원인 추적 어려움
  • 사용자 피드백 받기 전까지 큰 비용 지출
  • 롤백 비용 = 전부 폐기

작게 쪼개면 모든 측면이 개선된다.

실제 적용 (이 사이트)

오늘 작업이 한 번에 끝나지 않은 이유:

  • Phase 1: 라우팅·MDX 베이스
  • Phase 3: 인터랙티브 (검색·단축키·테마)
  • Phase 4: Editorial Modern 디자인
  • Phase 5: 누런 톤 박멸 (quality-gate 3 사이클)
  • Phase 7: 모바일 GNB
  • Phase 8: 모바일 quality-gate (3 사이클)
  • ...

각 Phase마다 빌드·push·라이브 반응 확인. 다음 Phase 결정.

만약 처음부터 "사이트 전체 리뉴얼"을 한 commit으로 했다면? "엄청 못생겼어" 피드백을 받는 시점에 되돌릴 수가 없다.

가이드

  • 한 commit = 한 가지 관심사
  • 한 PR = 머지 후 한 번에 검증 가능한 단위
  • 한 세션 = 1-3 commit (그 이상이면 다음 세션 분리)
  • 각 단계마다 빌드 통과 + 사용자 가시 동작

함정

  • "한 번에 끝내야 효율적"이라는 환상. 실제로는 작은 사이클이 누적 효율 더 큼.
  • 작게 쪼개느라 추상화 + 추가 작업 = 본 작업보다 큼. 작은 사이클의 비용 < 큰 사이클의 위험일 때만 적용.
  • "이건 작아서 작게 못 쪼개" — 대부분 사실 아님. 5분 이상 작업이면 쪼갤 자리 있음.

핵심

작게 배포하고 다시 읽는 것은 사용자 반응을 의사결정 입력으로 사용하는 패턴이다. 큰 commit은 본인 직관에만 의존한다.

관련

관련 노트

  • 🌳 Codex 경쟁 — 모든 작업을 두 번 굴려보기

    코딩·추론·리뷰 등 모든 작업에서 Codex와 Claude를 병렬 실행하고 결과를 교차 검증. 토큰 비용은 늘지만 사각지대가 줄어든다.

  • 🌱 AI 반복 위임의 mode collapse — 출력이 한 점으로 수렴한다

    AI에게 "더 만들어줘"를 반복하면 출력이 점점 비슷해진다. 모델이 고확률 영역으로 수렴하는 mode collapse다. 다양성은 저절로 안 나오고 명시적으로 강제해야 한다.

  • 🌳 dispatch — 의도 기반 라우팅 패턴

    모든 사용자 메시지를 3 에이전트 경쟁으로 최적 스킬에 라우팅. 키워드 매칭이 아닌 의미론적 의도 분석.

  • 🌱 AI 페어와 야간 incident 디버깅 — 새벽에 혼자가 아닌 워크플로우

    새벽 장애 대응의 진짜 비용은 디버깅 시간이 아니라 외로움과 결정 부담이다. AI 페어는 문법적 코드보다 의사결정 cross-check에서 가치가 크다.

  • 🌱 모호한 지시를 측정 기준으로 — "잘 해줘"를 실행 가능하게

    "전체적으로 개선해줘", "다양하게", "깔끔하게"는 실행자마다 다르게 해석된다. 작업 전에 모호한 품질 단어를 측정 가능한 기준으로 번역하면 재작업과 오해가 줄어든다.
