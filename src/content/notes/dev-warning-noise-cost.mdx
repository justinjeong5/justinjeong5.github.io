## 문제 — "나중에 볼게요" 워닝의 누적

dev 서버를 띄우면 매번 같은 warning이 출력된다.

```
warning - export `useEffect` not found in module @sentry/react
warning - export `Component` not found in module @sentry/react
warning - ...
```

처음 보면 신경 쓴다. 두 번째도 본다. 세 번째부터 무시한다. 한 달 후엔 warning이 50줄 출력되어도 스크롤만 한다.

문제는 그 노이즈 안에 **진짜 에러**가 섞일 때다. 새 warning이 추가됐는데 기존 warning들과 섞여서 안 보인다. 또는 runtime에서 발생하는 에러가 dev 로그에 한 줄 찍히는데 묻힌다.

## 좋은 알람의 4기준이 dev 로그에도

[좋은 알람 4기준](/notes/good-alert-four-criteria)이 dev 로그 warning에도 적용된다.

1. **사람이 행동할 일이 있는가** — warning이 떴는데 "지금 뭐 해?"가 명확하지 않으면 noise
2. **임계가 의미 있는가** — 라이브러리 내부 호환성 경고가 사용자 영향 0이면 진짜 신호 아님
3. **runbook이 있는가** — 이 warning이 떴을 때 어떻게 대응할지 어딘가에 있어야
4. **해결되거나 명시적 silence되는가** — 무시 누적은 시스템 신뢰 0

## 처방 — 3분류

각 warning을 셋 중 하나로 분류한다.

**A. 즉시 해결**: 진짜 문제. 라이브러리 업그레이드, 코드 수정으로 끝낸다.

**B. 명시적 silence**: 무시해도 되는 warning. 단, "그냥 무시"가 아니라 코드/config에 명시.

```js
// next.config.js
webpack(config) {
  config.ignoreWarnings = [
    {
      module: /node_modules\/@sentry\/react/,
      message: /export .* not found/,
    },
  ];
  return config;
}
```

이렇게 적어두면 다음부터 같은 warning이 안 보인다. **새 warning만 보인다.**

**C. 알려진 이슈로 link**: 라이브러리 자체 이슈인 경우 GitHub issue로 link 남기고 정리. silence와 함께 코멘트로 issue URL.

세 분류 다 안 한 채로 "그냥 둠"이 노이즈 누적의 원인.

## "기억해두자"는 안 됨

"이거 나중에 봐야지" → 한 달 후에도 그대로다. 인간 메모리는 휘발성이라.

처방은 두 가지:

- **즉시 해결 (A)**: 5분 안에 가능하면 지금
- **명시적 silence (B)**: 코드/config에 한 줄 박는다

둘 다 안 되면 새 warning이 보이지 않을 가능성 ↑.

## react import warning 패턴

자주 등장하는 케이스: 라이브러리가 React 17 export 형식으로 import하는데 React 19에서 named export 변경. 라이브러리 측 fix 대기 중이면:

- A 불가 (라이브러리 측이 안 고침)
- B 적절 (사용자 영향 0인 호환성 경고)
- C 권장 (issue link 남기고 fix 시 silence 제거)

silence는 임시 조치, **fix되면 즉시 silence 제거**가 원칙. 안 그러면 silence가 stale.

## 함정

- **silence 패턴이 너무 넓음**: `module: /node_modules/` 식으로 전체 무시 → 진짜 라이브러리 버그도 묻힘. 좁게 한정.
- **silence 후 잊음**: silence 코드 옆에 issue link·기한·회고 날짜를 코멘트로 남긴다.
- **새 warning 무감지**: silence를 자주 하면 "또 새 거 silence"가 습관. 새 warning은 한 번 검증 후 silence.
- **dev only라 production 영향 없다고 무시**: dev 로그 무신뢰는 일반 무신뢰로 이어진다. dev도 진지하게.

## CI에도 똑같이 — warning을 error로 승격

dev 로그뿐 아니라 CI 빌드 로그도 같은 누적이 일어난다. CI는 한 발 더 나아갈 수 있다.

- silence 결정한 warning → `ignoreWarnings`로 숨김
- 그 외 **새 warning은 CI 실패로 승격** (`--max-warnings 0`)

이러면 새 warning이 PR에서 빨간 불로 잡힌다. "나중에 봐야지"가 구조적으로 불가능해진다. 단, 승격 전에 기존 warning을 전부 분류(해결/silence/link)해야 한다. 안 그러면 CI가 항상 빨간 불.

## 핵심

> dev warning도 알람이다. 신호와 노이즈를 분리 안 하면 둘 다 잃는다.
