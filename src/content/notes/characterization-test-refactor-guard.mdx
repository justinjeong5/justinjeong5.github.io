## 문제 — 손대기 무서운 코드

한 함수에 입출력·부작용·분기가 엉켜 있다. 순수 로직을 별도 헬퍼로 떼고 싶은데, 떼는 순간 무엇이 깨질지 알 수 없다. 테스트가 없으니 리팩토링이 무섭고, 구조가 엉켜 있으니 테스트도 못 짠다. 닭과 달걀이다.

명세서가 있으면 "이대로 동작해야 한다"는 기준이라도 있다. 그런데 레거시에는 명세가 없다. "지금 무엇을 하는지"만 있고, 그게 맞는지는 아무도 모른다.

## 해법 — 지금 동작을 박제한다

순서를 뒤집는다. 리팩토링 **전에** 현재 동작을 그대로 고정하는 테스트를 먼저 깐다. 이걸 characterization 테스트라 부른다(저장된 출력 스냅샷과 비교하는 golden master는 이를 구현하는 대표적 기법이다).

characterization 테스트는 "맞는 동작"이 아니라 "지금 동작"을 검증한다. 코드를 호출해서 나온 출력을 그대로 기대값으로 박아둔다.

```python
# 1단계: 현재 동작을 캡처한다 (한 번만 실행해서 golden을 만든다)
# 키는 입력의 repr이 아니라 안정적 case id로 — 키 직렬화 비결정성에 의존하지 않는다
def capture_golden(cases):
    return {cid: repr(build_record(c)) for cid, c in cases}

# 2단계: 박제한 golden을 회귀 테스트로 고정한다
GOLDEN = load_golden("record_builder.golden.json")

@pytest.mark.parametrize("cid,case", SAMPLE_INPUTS)
def test_characterization(cid, case):
    assert repr(build_record(case)) == GOLDEN[cid]
```

이제 안전망이 생겼다. `build_record`의 내부를 어떻게 바꾸든, 출력이 golden과 한 글자라도 다르면 테스트가 빨개진다.

## 행위-보존 리팩토링의 검증 루프

순수 함수 추출처럼 "동작은 그대로, 구조만 바꾸는" 리팩토링에는 결정론적 검증 루프가 붙는다.

1. characterization 테스트를 먼저 깐다 (전부 초록).
2. 순수 로직을 헬퍼로 추출한다.
3. 테스트를 다시 돌린다.
4. 출력이 byte-identical이면 행위가 보존됐다는 기계적 증거가 된다.

"잘 리팩토링했어"는 사람이 눈으로 확인해야 한다. "출력이 골든과 동일하다"는 기계가 확인한다. 후자라야 안심하고 칼을 댄다.

개인 자동화에서도 같은 패턴을 썼다. 부동산 데이터 파이프라인의 큰 함수에 characterization 안전망을 먼저 두른 뒤, 레코드 빌더 다섯 종을 순수 헬퍼로 추출했다. 동작이 테스트로 고정돼 있으니 추출이 안전했다. ([로그](/logs#2026-06-04-pure-function-characterization))

## 좋은 입력 샘플 고르기

golden은 입력 샘플만큼만 강하다. 샘플이 happy path만이면 엣지에서 깨진 회귀를 못 잡는다.

- 정상 케이스 몇 개
- 빈 입력·null·빈 배열
- 경계값 (음수, 0, 최대치)
- 실제 운영 데이터를 익명화한 표본 (생성된 합성 데이터보다 현실 분포를 잘 담는다)

부작용 없는 순수 함수일수록 샘플링이 쉽다. 입력과 출력만 있으면 되니까. "characterization 테스트가 안 써진다"는 느낌은 보통 로직과 부작용이 안 떼어졌다는 구조 신호다.

## 함정

- **버그까지 박제한다**: characterization은 "맞는 동작"이 아니라 "지금 동작"을 고정한다. 현재 코드에 버그가 있으면 그 버그도 golden에 들어간다. 이건 의도된 거다 — 먼저 현재를 고정해 안전망을 만들고, **그다음에** 의도적으로 동작을 바꾼다. 동작을 바꾸는 커밋과 구조만 바꾸는 커밋을 절대 섞지 않는다.
- **출력이 결정론적이지 않다**: 타임스탬프·랜덤·정렬되지 않은 set은 같은 머신에서도 매 실행 출력이 바뀐다. 시간은 주입, 랜덤은 시드 고정, 컬렉션은 정렬해서 직렬화를 결정론적으로 만든 뒤 박제한다.
- **부동소수점을 매 실행 흔들리는 것과 헷갈린다**: 부동소수점은 위와 다른 부류다. IEEE 754 연산은 동일 플랫폼·런타임에서 완전히 결정론적이다(`0.1 + 0.2`는 매번 똑같은 `0.30000000000000004`다). 문제는 크로스-환경 재현성이다 — golden을 다른 플랫폼·런타임·컴파일 옵션(`-ffast-math`, 병렬 reduction 순서 등)에서 재생성하면 마지막 비트가 달라질 수 있다. 비교 시 허용 오차(tolerance)를 두거나 고정 정밀도로 직렬화해 박제한다.
- **golden을 무지성으로 갱신한다**: 테스트가 빨개지면 "golden을 새 출력으로 덮어쓰면 초록"이다. 이러면 안전망이 죽는다. golden 변경은 의도된 동작 변경일 때만, 리뷰에서 diff를 사람이 확인하고 갱신한다.
- **안전망을 영구 테스트로 착각한다**: characterization은 리팩토링용 임시 발판이다. 동작의 의도가 명확해지면 "왜 이 출력이 맞는가"를 검증하는 행위 기반 테스트로 승격시킨다. golden은 그때 걷어낸다.

## 핵심

> characterization 테스트는 "맞는 동작"이 아니라 "지금 동작"을 박제한다. 명세가 없어 "무엇이 맞는지" 모르는 레거시에서도, "무엇이었는지"는 고정할 수 있다 — 그게 안심하고 칼을 댈 안전망이다.
