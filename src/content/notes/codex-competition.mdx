## 왜 경쟁시키는가

단일 모델로 결과를 만들고 자기 검증하는 건 confirmation bias의 함정이다. 같은 모델이 만들고 같은 모델이 평가하면 같은 사각지대를 공유한다.

다른 모델(Codex)로 동일 작업을 병렬 수행하고 두 결과를 비교하면:

- 양쪽 모두 같은 결론에 도달하면 신뢰도 ↑
- 결론이 갈리면 어디서 갈리는지가 그 자체로 디버깅 단서
- 한쪽만 발견한 엣지 케이스는 거의 확실히 진짜 엣지 케이스

## 트레이드오프

**비용**: 토큰 두 배. 시간도 늘어남 (직렬이면 ×2, 병렬이면 max).

**가치**: 작업 종류에 따라 다르다. 빠른 단순 코드는 경쟁 가치 ↓. 설계 결정·복잡한 리팩토링·보안 리뷰는 경쟁 가치 ↑.

## 실용적 규칙

- **경쟁이 기본값**: 설계·리뷰·복잡한 리팩토링은 항상 두 번 굴린다.
- **단순 작업은 예외**: 한 줄 수정·오타·뻔한 fetch는 경쟁 가치가 낮으므로 단독.
- **사용자 명시 시 단독**: "코덱스 없이"라 하면 신뢰가 충분한 신호로 받아들임.
- **경쟁 결과 종합 방법**:
  1. 두 결과 다 read
  2. 일치하는 부분 채택
  3. 갈리는 부분만 사람이 결정 (또는 더 깊이 분석)

## 함정

- 두 모델이 같은 학습 데이터 기반이면 같은 편향 공유 가능. 진짜 다른 관점이 아닐 수 있다.
- 종합 단계에서 "둘 다 비슷하니까 OK"로 빠지면 경쟁의 의미가 사라짐. **일치 자체가 검증은 아님**, 일치하는 *근거*가 검증.
