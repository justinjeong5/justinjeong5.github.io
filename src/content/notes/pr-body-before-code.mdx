## 흔한 PR 본문

```
fix typo
```

또는

```
WIP
```

또는 빈 본문.

이게 PR 100개 중 70개의 현실입니다. 그러면 리뷰어가 코드를 보면서 "이게 무슨 의도지?"를 추측해야 합니다. 추측이 틀리면 잘못된 리뷰가 갑니다.

## 잘 쓴 PR 본문 구조

```markdown
## Why
무엇을 해결하려고 한 PR인가. 1-2 문장.

## What changed
주요 변경 3-5개 bullet. 코드 라인 인용 없이 의도만.

## Trade-off
선택한 방법과 버린 방법. 왜 이 방법인가.

## Test plan
어떻게 검증했는가. 자동 테스트 + 수동 시나리오.

## Risk
머지 후 무엇이 깨질 수 있는가. 롤백 방법.
```

이렇게 적는 데 5분 정도 걸립니다. 리뷰어는 본문 보고 어디부터 코드 읽을지 결정할 수 있습니다. 본문이 없는 PR보다 30분 빠른 머지.

## 누가 가장 큰 이득인가

**6개월 후의 자기 자신**. PR 본문이 commit message보다 더 풍부합니다. `git blame` → PR 링크 → 본문 → "아, 이 결정의 근거가 이거였구나."

코드만 보면 절대 안 보이는 컨텍스트가 PR 본문에 살아남습니다.

## 함정

- PR 본문에 코드 인용 너무 많으면 본문이 코드 중복. 의도만 적기.
- "Why" 없이 "What"만 적으면 가치 ↓. 다 알지 What. 결정의 이유를 묻는 게 Why.
- "TODO: 본문 나중에" → 영원히 안 적음. PR 만들 때 같이.

## 핵심

> PR 본문은 코드를 읽는 사람에게 **무엇을 읽어야 하는지**를 미리 알려주는 안내문이다.
