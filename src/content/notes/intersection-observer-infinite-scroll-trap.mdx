## 왜 IntersectionObserver인가

스크롤 이벤트 + `getBoundingClientRect`로 만들면 매 프레임 layout reflow가 일어난다. 페이지가 길어질수록 스크롤이 점점 무거워진다.

IntersectionObserver는 브라우저가 비동기로 가시성을 계산해서 callback을 던진다. 메인 스레드 부담이 없고 60fps가 유지된다.

그래서 무한 스크롤의 디폴트는 이제 IntersectionObserver다. 그런데 매번 같은 함정에 빠진다.

## 함정 1 — observer를 매 렌더마다 새로 만든다

```jsx
function InfiniteList({ onReachEnd }) {
  const ref = useRef();

  // 매 렌더마다 새 observer 인스턴스
  const observer = new IntersectionObserver((entries) => {
    if (entries[0].isIntersecting) onReachEnd();
  });

  useEffect(() => {
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, []);
  // ...
}
```

매 렌더마다 observer가 새로 만들어지는데 effect는 `[]`라 처음 것만 잡힌다. callback이 stale closure가 된다.

**고쳐**: observer를 useEffect 안에서 만들거나 useRef로 보관.

```jsx
useEffect(() => {
  const observer = new IntersectionObserver((entries) => {
    if (entries[0].isIntersecting) onReachEnd();
  });
  if (ref.current) observer.observe(ref.current);
  return () => observer.disconnect();
}, [onReachEnd]);
```

## 함정 2 — rootMargin 누락

기본 rootMargin은 `0px`다. 즉, 센티넬 요소가 뷰포트에 **완전히 진입**해야 callback이 발화한다.

이러면 사용자가 끝까지 스크롤한 후에야 다음 페이지 로드가 시작된다. 사용자는 빈 화면을 본다.

**고쳐**: rootMargin을 미리 줘서 화면 밖에서 미리 로드.

```jsx
new IntersectionObserver(callback, {
  rootMargin: '200px 0px', // 위아래 200px 더 일찍 발화
});
```

목록 항목 높이의 1.5-2배 정도가 적정. 너무 크면 불필요한 fetch, 너무 작으면 사용자가 빈 화면을 본다.

## 함정 3 — threshold=0의 의미

```jsx
new IntersectionObserver(callback, { threshold: 0 });
```

`threshold: 0`은 "1픽셀이라도 보이면 발화"가 아니다. **0% 통과 시 발화** — 정확히 경계가 닿는 순간. 1픽셀 차이로 발화 안 할 수 있다.

비슷하게 `threshold: 1`은 "100% 다 보일 때". 작은 요소는 OK, 뷰포트보다 큰 요소는 **영원히 발화 안 함**.

**고쳐**:
- 무한 스크롤 센티넬은 보통 작은 요소 → `threshold: 0` 또는 `threshold: 0.1` 정도
- 큰 카드의 가시성 감지는 `threshold: [0, 0.25, 0.5, 0.75, 1]` 같이 배열로

## 함정 4 — observe 해제 시점

리스트 끝에 도달해서 더 이상 페이지가 없으면 `unobserve()`해야 한다. 안 하면 hasNextPage가 false인데도 callback이 계속 발화한다.

```jsx
const observer = new IntersectionObserver((entries) => {
  if (entries[0].isIntersecting) {
    if (hasNextPage) {
      onReachEnd();
    } else {
      observer.unobserve(entries[0].target);
    }
  }
});
```

또는 더 안전하게 IntersectionObserver를 hasNextPage가 false가 된 시점에 disconnect하고, useEffect deps에 hasNextPage를 넣는다.

## 보너스 — root 옵션의 함정

root를 명시하지 않으면 뷰포트가 root다. 컴포넌트가 **스크롤 컨테이너 안**에 있으면 root를 그 컨테이너로 지정해야 한다.

```jsx
new IntersectionObserver(callback, {
  root: scrollContainerRef.current,
  rootMargin: '200px 0px',
});
```

root를 안 주면 컨테이너 안에서 스크롤해도 뷰포트 기준으로 계산돼서 영원히 발화 안 한다.

## 체크리스트

- [ ] observer를 useEffect 안에서 만들었는가
- [ ] rootMargin을 줘서 화면 밖에서 미리 발화하게 했는가
- [ ] threshold 값의 의미를 정확히 알고 있는가
- [ ] hasNextPage가 false일 때 unobserve 또는 disconnect하는가
- [ ] 스크롤 컨테이너가 별도면 root를 지정했는가

## 핵심

> IntersectionObserver는 **메인 스레드 부담 없는 가시성 감지**의 정답이지만, 옵션 의미를 정확히 모르면 발화 안 하거나 무한 발화한다. 4가지 함정만 피하면 60fps 무한 스크롤이 된다.
