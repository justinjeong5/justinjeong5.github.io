## 문제 — "있어도 그만"이 아니다

새 패키지 시도하다가 안 쓰게 됐을 때, package.json에서 빼는 걸 잊는다. 또는 코드가 다른 라이브러리로 옮겨갔는데 옛 의존성을 그대로 둔다.

"어차피 import 0건인데 뭐가 문제냐."

문제는 누적된다.

- `pnpm install` 시간 ↑ (특히 CI 매번)
- lockfile 부피 ↑ (PR diff에서 노이즈)
- transitive 의존성이 따라 들어옴 (직접 안 쓰는데 그 의존성의 의존성까지)
- transitive CVE 표면적 ↑ (rollup·node-fetch·ajv 등)
- supply chain attack 노출 ↑ (npm package compromise → 영향)

가장 큰 비용은 **transitive CVE 보고**다. 매주 audit 돌리면 "직접 안 쓰는 라이브러리"의 취약점이 계속 잡힌다. "직접 의존 아니라 영향 낮음"이라고 분류하고 정리는 매번 미룬다.

## 해법 — 사용처 0 검증 + 즉시 제거

정기적으로 사용처 0건인 의존성을 식별한다.

```bash
pnpm depcheck            # missing/unused 후보
npx knip                 # import 그래프 기반 unused 탐지
npx ts-prune             # ts 미사용 export
```

자동 도구 결과를 그대로 믿지 않는다. **수동 검증** 필수.

- 동적 import (`import('pkg')`)는 도구가 못 잡음
- `next.config.js`·`vite.config.js`·`postcss.config.js` 등 설정 파일에서만 사용
- peer dependency로만 필요한 케이스
- 빌드 후크/CLI 스크립트에서만 사용

검증 후 제거 → `pnpm install` → 빌드 → 테스트 통과 확인. 한 PR에 한 패키지씩이 안전 (rollback 단위 일치).

## init 코드가 진짜 신호

특히 `@sentry/nextjs`·`@datadog/browser-rum` 같은 observability 라이브러리는 **import만으로 동작하지 않는다**. `sentry.client.config.ts`·`instrumentation.ts` 같은 초기화 파일이 있어야 실제 동작한다.

다음 조건이면 dead:

- package.json에 있음
- 코드에 `import` 0건
- init 설정 파일 0건

→ **제거 안전**. 도구는 이 케이스를 못 잡는다. observability 라이브러리는 효과를 import로 안 보여주니까. **init 파일 존재 확인이 진짜 신호.**

## 함정

- **peer dep만 안 쓰는 것 같음**: 다른 패키지의 peer로 필요. `pnpm why <pkg>`로 확인.
- **CLI 스크립트만 사용**: `package.json scripts`에서만 호출. import 검색으로 안 잡힘.
- **빌드 후크 사용**: webpack/vite/next plugin이 내부적으로 사용. config 파일 확인.
- **한꺼번에 정리**: 5개 패키지를 한 PR에 → 회귀 시 어느 게 원인인지 모름. 1개씩.
- **dev dependency 무시**: dev도 install·lockfile에 영향. 똑같이 정리 대상.
- **"transitive라 영향 낮음" 회피**: 회피가 누적되면 정리할 패키지가 10개+. 직접 의존 정리가 transitive도 같이 줄인다.

## supply chain이 진짜 위험

4가지 비용 중 가장 과소평가되는 게 supply chain 표면적이다. 안 쓰는 의존성도 `npm install`에 포함되어 빌드 머신·CI에서 **install 시점에 코드를 실행**할 수 있다(postinstall script).

2021년 이후 npm 생태계의 공급망 공격은 대부분 이 경로다.

- 인기 패키지가 compromise → 악성 postinstall → CI secret 탈취
- 직접 안 쓰는 transitive 의존성이어도 install되면 노출

→ 안 쓰는 직접 의존성을 제거하면 그 transitive 트리 전체가 install에서 빠진다. dead dependency 정리는 코드 위생이 아니라 **보안 작업**이다. `--ignore-scripts` install + 명시적 allowlist도 같이 검토.

## 핵심

> 안 쓰는 의존성은 **install·lockfile·CVE·supply chain** 4가지 비용을 동시에 만든다. 코드 정리만큼 정기적이어야 한다.
