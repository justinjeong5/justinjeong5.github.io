## SameSite의 핵심 질문

쿠키 SameSite 속성은 한 질문에 답한다.

> 이 쿠키를 **다른 사이트에서 우리 사이트로 가는 요청**에 함께 보낼 것인가?

이게 cross-site / 3rd party context의 정의다. SameSite의 세 값은 이 질문에 다른 답을 한다.

## SameSite=Strict — 보낸 적이 없다

cross-site 요청에 **절대 쿠키 안 보냄**.

```
Set-Cookie: session=abc123; SameSite=Strict
```

다른 사이트에서 우리 사이트 링크를 클릭해서 들어와도 쿠키 안 보낸다. 즉, 사용자가 외부에서 들어오는 첫 요청에는 로그인 세션이 없다. 우리 사이트 안에서 다시 클릭해야 쿠키가 붙는다.

**사용 케이스**: 매우 민감한 작업(은행 송금, 비밀번호 변경). 일반 로그인 세션에는 과함.

## SameSite=Lax — top-level navigation은 OK

cross-site 요청 중 **사용자가 명시적으로 우리 사이트로 이동하는 경우**에만 쿠키 보냄.

구체적으로:
- ✅ 다른 사이트에서 우리 사이트 링크 클릭 (top-level navigation, GET)
- ✅ 주소창에 우리 사이트 URL 직접 입력
- ❌ 다른 사이트의 iframe에서 우리 사이트 호출
- ❌ 다른 사이트가 우리 사이트로 POST/PUT/DELETE
- ❌ 다른 사이트의 `<img src>` 같은 sub-resource 요청

```
Set-Cookie: session=abc123; SameSite=Lax
```

대부분의 일반 로그인 세션은 Lax가 적절. CSRF 공격(다른 사이트가 우리 사이트로 POST)을 막으면서 사용자 흐름은 자연스럽다.

**현재 브라우저 디폴트**: SameSite 미지정 쿠키는 Chrome/Firefox/Safari가 자동으로 Lax 취급.

## SameSite=None — 모든 cross-site에 보냄 (Secure 필수)

cross-site 요청 종류 무관하게 **항상** 쿠키 보냄. 단, 반드시 `Secure` 같이.

```
Set-Cookie: session=abc123; SameSite=None; Secure
```

`Secure` 없으면 브라우저가 쿠키 자체를 거부한다.

**사용 케이스**: 진짜 cross-site 통신이 필요한 경우.
- iframe 안에서 동작하는 위젯 (chat·결제·comments)
- 다른 도메인의 API를 호출하는 SPA (CORS + cookies)
- 광고 SDK / 트래킹

## 3rd party cookie deprecation의 의미

Chrome이 3rd party 쿠키를 차단하는 흐름은 SameSite=None 쿠키가 **결국 안 보내진다**는 의미다. SameSite=None은 cross-site 쿠키이고 그게 곧 3rd party 쿠키니까.

영향:
- iframe 위젯에서 자체 도메인 쿠키로 세션 유지가 안 됨
- 광고 트래킹에 사용자 식별 어려움
- federated identity(SSO) 흐름 일부 깨짐

대응:
- **First-party set 등록**: 같은 회사 도메인을 한 묶음으로 등록 (Chrome First-Party Sets)
- **CHIPS (Partitioned Cookies)**: `Set-Cookie: ...; Partitioned; SameSite=None; Secure` — 호출한 top-level 사이트별로 쿠키 분리
- **iframe 대신 popup**: top-level navigation으로 우회

## 무엇을 골라야 하나

| 시나리오 | 추천 |
|---|---|
| 일반 로그인 세션 | `Lax` (또는 미지정 → 자동 Lax) |
| 송금·비밀번호 변경 같은 민감 작업 | `Strict` 별도 쿠키 |
| iframe 위젯 / cross-site API | `None; Secure` (deprecation 대비 CHIPS) |
| CSRF 토큰 | `Strict` (top-level navigation에서도 쿠키 안 가도 됨) |

## 함정

- **Strict로 일반 세션 만들기**: 사용자가 외부 링크로 들어오면 로그아웃 상태처럼 보임. UX 손상.
- **None만 쓰고 Secure 안 붙임**: 브라우저가 쿠키 자체를 무시함. HTTP 환경에서 디버깅하다 빠지기 쉬운 함정.
- **3rd party cookie deprecation을 무시**: iframe 위젯이나 cross-domain SPA가 어느 날 갑자기 동작 안 함. 사전 마이그레이션 (CHIPS, First-Party Sets) 필요.
- **CSRF 토큰을 Lax 쿠키로**: Lax는 top-level GET에서 쿠키를 보낸다. CSRF 토큰을 Lax로 두면 GET 기반 CSRF 일부 시나리오에서 취약. Strict 권장.
- **개발 환경(localhost)에서 Secure 누락**: Secure는 HTTPS 필수인데 localhost는 예외 처리(브라우저별 차이). 프로덕션에서 처음으로 Secure 누락 문제 발견하는 경우.
- **Lax 디폴트 가정을 모든 브라우저에 적용**: Safari·Firefox·구버전 Chrome 동작이 살짝 다르다. 명시적으로 SameSite 값을 적는 게 안전.

## 핵심

> SameSite는 **CSRF 방어 + 사용자 흐름**의 균형을 결정한다. 대부분 Lax가 정답이고, iframe·cross-site는 None+Secure+Partitioned, 민감 작업은 Strict 별도 쿠키. 3rd party cookie deprecation 시대엔 None 쿠키를 의식적으로 줄여야 한다.
