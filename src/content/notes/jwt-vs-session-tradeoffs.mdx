## 두 모델

- **Session**: 서버에 세션 저장 (Redis·DB). 클라이언트는 session ID만 가짐.
- **JWT**: 토큰 자체에 payload를 담아 서버 스토리지 없음. 서명으로 무결성 검증.

JWT가 "stateless"라는 이유로 자주 기본값으로 선택되지만, 5가지 차원에서 보면 session이 더 단순한 경우가 많다.

## 차원 1 — 스토리지

| | Session | JWT |
|---|---|---|
| 서버 | session store 필요 (Redis) | 없음 |
| 클라이언트 | session ID (짧음) | 토큰 (보통 500바이트+) |

서버 단순화는 JWT 승. 단, 매 요청마다 클라이언트가 큰 토큰을 보내야 함.

## 차원 2 — 만료

**Session**: 서버가 만료 결정. session store에서 expire.

**JWT**: 토큰 안에 `exp` claim. 검증 시 확인. 만료 후 새 토큰이 필요.

**문제 — JWT는 만료 단축이 어렵다**:
보안 사고 시 "모든 토큰 즉시 만료" 가 필요한데, JWT는 발급된 토큰의 exp를 바꿀 수 없다. server-side blacklist 또는 짧은 exp + refresh token이 필요.

세션은 즉시 만료 가능 (서버에서 삭제).

## 차원 3 — Revocation

가장 큰 차이.

**Session**: 즉시 revoke (서버에서 삭제). 다음 요청부터 거부.

**JWT**: 토큰 자체로는 revoke 불가. 옵션:
- **Blacklist**: revoke된 JWT를 서버에 저장 → stateless 이점 사라짐
- **짧은 exp + refresh token**: 5-15분 access token + refresh token으로 갱신. revoke은 refresh token만 막음.
- **rotating refresh token**: refresh 마다 새 refresh 발급, 옛 거 invalidate

JWT의 stateless는 revocation을 포기하거나 stateful blacklist를 더한 hybrid가 된다. **순수 stateless JWT는 보안 사고에 약함**.

## 차원 4 — 권한 정보 갱신

**Session**: 매 요청마다 서버에서 권한 조회. 권한 변경 즉시 반영.

**JWT**: 토큰 안에 권한 claim. 권한 변경되어도 토큰은 옛 권한 유지. 새 토큰 발급까지 stale.

**문제 사례**:
- 직원 퇴사 → 권한 제거 → 그러나 JWT는 만료까지 권한 유지 → 보안 구멍

해결: JWT를 짧게 + 매 요청에서 권한 재조회 → JWT의 stateless 이점 사라짐.

## 차원 5 — 분산 시스템

**Session**: session store가 SPOF. Redis cluster 필요.

**JWT**: 서명만 검증하면 됨. 분산 검증 쉬움.

진짜 분산 환경(여러 서비스가 같은 토큰 검증)에선 JWT가 명확히 유리.

## 실용 선택 가이드

```
사용 사례
  ↓ 단일 서버 또는 같은 인프라 안의 서비스 수개?
  → Session (Redis) — 단순, revoke 즉시
  
  ↓ 여러 마이크로서비스 + 외부 파트너 API?
  → JWT (짧은 exp + refresh) — 분산 검증

  ↓ 모바일 앱 + 긴 로그인 유지?
  → JWT (refresh token) — 모바일은 cookie 다루기 어려움
```

## 흔한 misuse

- **JWT를 stateless라 부르며 blacklist 도입**: stateless 이점이 사라진다. blacklist 필요하면 session이 더 단순.
- **JWT 만료를 30일로 길게**: revoke 못 하는 토큰을 30일 유효? 사고 시 30일간 노출. exp는 15분-1시간.
- **JWT를 localStorage에**: XSS 시 토큰 탈취. HttpOnly cookie가 안전.
- **Session ID를 URL에**: 로그·referrer로 노출. cookie만 사용.
- **refresh token rotation 안 함**: 한 번 탈취된 refresh token이 영원히 유효.

## 함정

- "JWT가 더 modern" 같은 패션 선택: 5가지 차원을 비교 안 한 선택은 보통 session이 더 맞음.
- HttpOnly cookie + Secure + SameSite를 안 쓰면 두 모델 모두 취약.
- access token + refresh token을 둘 다 localStorage에 두면 XSS 시 무의미.
- CSRF 보호 누락: cookie 기반이면 CSRF 토큰 필수.

## 핵심

> JWT의 stateless는 **분산 검증**에선 명확한 가치지만, **revocation·권한 갱신**이 필요하면 stateless 이점이 사라진다. 단일 인프라면 session이 더 단순. JWT를 쓰려면 짧은 exp + refresh + rotation까지 같이 와야 함.
