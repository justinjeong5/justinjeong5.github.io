## 문제

테마가 둘 이상이면 같은 영역의 색이 라이트/다크에서 달라야 한다. 그래서 다음 같은 코드가 나오기 쉽다.

```css
.glass-bg {
  background: rgba(244, 241, 232, 0.92); /* 라이트 cream */
}
:root[data-theme='dark'] .glass-bg {
  background: rgba(20, 23, 26, 0.92); /* 다크 charcoal */
}
```

문제: 하드코딩 rgba 두 벌. 토큰 변경 시 두 곳 모두 수정. cascade 충돌 위험.

## 해결 — color-mix

```css
.glass-bg {
  background: color-mix(in srgb, var(--bg) 92%, transparent);
}
```

`--bg`가 라이트에선 white, 다크에선 zinc-950이면 한 줄로 두 테마 자동 처리.

## 활용 패턴

- **글래스 효과**: `color-mix(in srgb, var(--bg) 88%, transparent)`
- **그리드 라인**: `color-mix(in srgb, var(--text) 5%, transparent)`
- **status pill 배경**: `color-mix(in srgb, var(--accent-green) 14%, var(--bg-alt))`
- **다크 박스 위 부드러운 텍스트**: `color-mix(in srgb, var(--text-inverse) 82%, transparent)`

## 브라우저 지원

Safari 16.4+, Chrome 111+, Firefox 113+. 2024년 이후 모든 모던 환경. 구형 fallback 필요 시 PostCSS로 자동화.

## 단일 source 회복

한 디자인 리팩토링 사이클에서 `시각 일관성` 점수가 2/5에서 5/5로 올라간 결정적 변경이 hex/rgba 30+곳을 토큰 + color-mix로 일괄 치환한 것이었다. 토큰 하나만 바꾸면 라이트·다크가 자동으로 따라온다.

## 처음에 헤맨 것

처음엔 토큰을 정의만 해두고 기존 하드코딩 hex를 그대로 뒀다. themes.css에 다크 override를 추가했지만 cascade 순서 때문에 light 모드에서 cream 잔재가 남아 누런 톤이 빠지지 않았다. specificity 동률 + import 순서 의존 = 디버깅 지옥. 토큰 정의만으로는 부족하고 **사용처를 모두 토큰화**해야 single source가 된다는 걸 그때 배웠다.

## 함정

- `color-mix` 안에서 또 `color-mix`를 nested 호출하면 가독성 ↓
- `transparent`로 합성할 때 alpha 강도가 사람 직관과 다를 수 있음 — 14%는 거의 보이지 않는다, 50%는 의외로 진하다
- 셀렉터 specificity가 동일하면 import 순서 의존 → CSS 파일 import 순서 관리 필수
