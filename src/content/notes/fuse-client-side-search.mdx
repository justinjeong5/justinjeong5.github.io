## 문제

블로그·문서·디지털 가든은 검색이 핵심 인터랙션. 그러나:
- Algolia 같은 SaaS는 비용·세팅 부담
- 자체 검색 서버는 정적 사이트 정신 위반
- 브라우저 `Ctrl+F`는 페이지 안만 검색

## 해법 — Fuse.js

```js
import Fuse from 'fuse.js';

const docs = [...getAllCases(), ...getAllNotes(), ...getAllEssays()];

const index = new Fuse(docs, {
  keys: [
    { name: 'title', weight: 0.6 },
    { name: 'summary', weight: 0.3 },
    { name: 'topics', weight: 0.1 },
  ],
  threshold: 0.4,
  ignoreLocation: true,
});

const results = index.search('dispatch').map(r => r.item);
```

- 빌드 시 모든 콘텐츠 frontmatter 인덱싱 (본문은 옵션)
- 퍼지 매칭 (오타 허용)
- 가중치 지정 가능 (제목 중요, 토픽 보조)
- 번들 사이즈 ~7KB gzipped

## 한계

- 콘텐츠 100개 ~ 1000개까지 무난. 그 이상은 FlexSearch 또는 Pagefind 검토.
- 본문 전체 인덱싱하면 번들 ↑. frontmatter만 권장.
- 한국어 토큰화 약함. 영어·기호 위주 사이트에 더 적합.

## 인터랙션 패턴

Cmd+K 글로벌 검색 모달 + 결과 그룹별 (Cases·Notes·Essays·Logs) + 키보드 네비 (↑↓ Enter Esc) = 거의 모든 정적 사이트의 표준 패턴.

## 함정

- 인덱스를 매 검색마다 재생성하면 lag. **모듈 스코프에 cache**.
- HMR에서 stale index 위험. dev에서 페이지 새로고침 권장.
- 가중치(weight)는 도메인별 다름. 디지털 가든은 title > summary > topics, 코드 문서는 title > content 등.

## 함의

정적 사이트에 검색을 안 넣는 가장 큰 이유는 보통 "복잡할 것 같아서"이다. Fuse.js는 그 직관을 뒤집는다.
