## 문제 — 즉시 시작이 재작업을 부른다

복잡한 작업을 받자마자 코드부터 만지면 흔한 패턴이 일어난다.

- 50% 만들고 보니 접근법이 잘못됨
- 80% 만들고 보니 사용자가 원한 건 다른 거였음
- 100% 만들고 보니 컨벤션이 어긋남

재작업 비용 = 처음 작업 시간의 50~150%. 토큰도 같은 비율로 든다.

## 해법 — Plan Mode 먼저

Claude Code의 **Plan Mode**: `Shift+Tab`으로 진입. 이 모드에서는 코드 수정·파일 생성·외부 효과 없는 도구만 사용 가능. **계획만 짠다.**

계획 항목:
- 변경할 파일 목록
- 각 파일의 변경 내용 요약
- 기존 컨벤션 확인 결과
- 예상 트레이드오프
- 검증 방법

계획이 나오면 **사용자가 검토**. 방향 다르면 여기서 수정. 코드는 아직 0줄.

## Auto 전환 → 1-shot

계획 확정 후 Auto 모드 전환 (Plan Mode 종료). 확정된 계획대로 **1-shot 구현**한다.

장점:
- 중간 방향 변경 0 → 재작업 0
- 계획이 명확하니 구현 속도 ↑
- 토큰 효율 ↑ (불필요한 탐색 없음)

체감: 5분 이상 작업에서 **재작업 30~50% 절감**.

## 언제 쓰나

- 변경 파일 3개 이상
- 기존 코드 컨벤션 파악 필요
- 트레이드오프가 있는 결정
- 사용자가 의도를 잘 못 표현했을 가능성

작은 작업(1파일, 명백한 한 줄 수정)은 오히려 느림. 계획 짜는 비용 > 즉시 구현 비용.

## 계획의 구체성

가장 흔한 실패는 계획이 **너무 추상적**일 때.

❌ "X 모듈 추가하고 테스트 추가, Y 페이지에 연결"
✅ 파일 경로·함수 이름·시그니처 수준:
```
src/lib/payment.ts
  - export async function chargeCustomer(id: string, amount: number): Promise<Receipt>
src/pages/checkout.tsx
  - useEffect 안에 chargeCustomer 호출 추가
tests/payment.test.ts
  - 정상/실패/네트워크 에러 3 케이스
```

이 수준이면 Auto 1-shot이 헤매지 않는다.

## 함정

- **계획이 너무 추상적**: 구현 시 또 헤맴. 파일·함수·시그니처 수준까지.
- **계획 확정 안 하고 Auto**: "대충 이거 같은데" 상태로 Auto 가면 재작업 발생. 사용자 명시 승인 후 Auto.
- **모든 작업에 Plan Mode**: 작은 작업도 매번 Plan → 오히려 느림. 5분 임계.
- **계획 검토 없이 통과**: 사용자가 "OK"만 하고 검토 안 함. Plan Mode 가치 0. 계획을 정독해야 함.

## spike와의 순서

미지의 영역이면 Plan Mode 전에 spike가 먼저다. 모르는 걸 계획할 수 없다.

```
미지의 영역?
  ├─ 예 → spike (버릴 코드로 학습) → Plan Mode (학습 위에 계획) → Auto 1-shot
  └─ 아니오 → Plan Mode → Auto 1-shot
```

spike 없이 모르는 영역을 Plan Mode로 계획하면, 계획 자체가 틀린 가정 위에 선다. spike가 사실을 주고, Plan Mode가 그 사실로 계획을 짜고, Auto가 1-shot 구현한다.

## 핵심

> 코드를 빨리 쓰는 것보다 **다시 쓰지 않는 것**이 더 빠르다.
