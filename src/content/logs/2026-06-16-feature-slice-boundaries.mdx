## 맥락

토스의 기차 예매 흐름을 화면 그대로 따라 만드는 개인 클론을 했다. 화면은 원본을 보고 베끼면 되니 고민할 게 없다. 진짜 고민은 그 화면 뒤의 구조였다 — 검색 폼, 역 선택, 티켓 목록, 예매 확인이 한 컴포넌트 트리에 뭉치면 처음엔 빠르지만 곧 서로의 상태를 들춰보기 시작한다.

처음부터 완벽한 모듈 경계를 그리려 하지 않았다. 경계가 어디인지는 만들기 전엔 잘 안 보인다. 그래서 일단 동작하게 만든 뒤, 중복이 보이거나 한 곳이 남의 사정을 너무 많이 알기 시작하면 그때 잘랐다.

## 한 일

기능을 수직으로 잘라 나갔다. 검색·역 선택·티켓 선택·예매 확인을 각각 feature로 떼고, 그 안의 상태를 feature별 훅으로 모았다 — 검색 폼은 `useTrainSearchForm`이 날짜·승객 수 선택 같은 하위 훅을 조립하고, 역 선택은 `useStationSelect`가 들고 있다. 구조는 features 디렉터리와 shared 디렉터리로 나뉘는 수직 슬라이스(Feature-Sliced) 모양으로 수렴했다.

- **도메인 키를 훅 뒤로 숨겼다**: 페이지는 쿼리 파라미터 이름이 아니라 도메인 언어로 말하게 했다. 왕복·편도 파싱과 검색 폼의 내부 키를 훅 안으로 넣으니, 페이지는 "무엇을"만 알고 "어떻게 직렬화되는지"는 모른다.
- **API 경로를 호출하는 곳 옆에 뒀다**: 여러 곳이 공유하는 URL 계약은 shared routing에 모으고, 한 기능만 쓰는 예매 API 경로는 그 feature 안에 뒀다. 규칙은 하나였다 — 계약은 공유되는 곳에, 호출은 기능 옆에.
- **교차로 반복되는 조각만 공유 섹션으로 추출했다**: 예매 요약·상세·티켓 섹션, 승객 수 행처럼 여러 화면에 같은 모양으로 나오는 것만 골라 공유 컴포넌트로 뺐다. 한 번만 쓰이는 건 굳이 올리지 않았다.
- 레이어를 가로지르는 import는 절대경로로 통일하고, 페이지 테스트는 "이 페이지가 책임지는 것"에만 초점을 맞췄다.

## 결정 — 디테일은 그것을 아직 소유하는 가장 작은 범위로 민다

경계를 가를 때 따른 기준은 한 줄이었다. "디테일을 그것을 소유하는 가장 작은 범위로 민다." 도메인 키는 페이지가 아니라 훅이 소유한다. 기능 전용 엔드포인트는 전역 라우팅이 아니라 그 feature가 소유한다. 반대로 여러 기능이 공유하는 URL 계약을 한 기능 안에 두면 거짓 소유가 되니, 그건 shared로 올린다.

테스트도 같은 기준이었다. 리팩토링으로 내부 구조가 계속 흔들렸는데, 페이지 테스트를 "페이지의 책임"에만 묶어 두니 추출·이동에도 테스트가 안 깨졌다. 구현 디테일에 묶인 테스트였다면 자를 때마다 빨개졌을 거고, 그럼 리팩토링이 무서워진다.

## 회고

클론은 베끼기 연습 같지만, 실은 구조 연습이다. 화면이 정답으로 주어져 있으니 변수는 구조 하나뿐이고, 그래서 모듈 경계를 깨끗하게 시험할 실험실이 된다. 원본을 충실히 옮길수록 "내가 더한 건 구조밖에 없다"가 선명해진다.

남은 규칙은 그 한 줄이다 — 모든 디테일을 그것을 아직 소유하는 가장 작은 범위로 민다. 도메인 키는 훅으로, 기능 엔드포인트는 feature로, 공유 계약은 shared로. 경계는 처음에 그리는 게 아니라, 누가 무엇을 아는지를 보고 한 칸씩 좁혀 발견하는 거였다.
