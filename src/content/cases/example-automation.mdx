## 문제

운영 정보가 여러 곳에 흩어지면 중요한 변화가 늦게 보이고, 반복 확인이 개인의 기억에 의존하게 됩니다.
"오늘은 봐야 할 것을 봤나"라는 불안이 매일 누적되었습니다.

## 의사결정

**버린 선택지**

- 모든 정보를 한 화면 대시보드로 — 만들기는 쉽지만 매일 들어가서 봐야 하는 부담은 그대로.
- 단순 cron 알림 — 실패 시 추적이 어렵고 중복 발송 위험.

**채택**

- 원본 데이터와 해석을 분리해 모바일 카드 포맷으로 구성.
- 중복 발송·실패 로그를 별도 저장 → "왜 안 왔지"를 1분 안에 추적.
- 스케줄 실행 후 dry-run 또는 로그 확인을 검증 루프로 포함.

- (추가 예정) 실패·복구 로그 구조 다이어그램
- (추가 예정) 전후 운영 시간 비교

## 임팩트

- 채워 넣을 때 중심 지표: **"얼마나 덜 확인하게 됐는지"**, **"어떤 판단이 빨라졌는지"**.
- 부수 효과: 신규 입사자 온보딩 시 운영 맥락 공유 자료로 활용 가능.

## 남은 보강 포인트

- 공개 가능한 화면 캡처와 전후 비교 수치 확보.
- 실패 패턴 3개를 분류해 다음 자동화 설계에 반영.
